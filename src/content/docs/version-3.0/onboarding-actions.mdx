The **Interactions** panel lets you define how flow elements respond to user events — such as taps, element appearances, and form submissions. For each event, you assign one or more actions: navigate between screens, show or hide elements, open URLs, set variables, and more. Use conditions to vary the response based on user data.

Each interaction follows a three-part chain:

1. **Element**: The screen component that starts the interaction — a button, quiz answer, input field, or anything else.
2. **Trigger**: The event that activates the logic, such as a tap, an element appearance, or a form submission.
3. **Action**: The task the flow performs in response. A single trigger can run multiple actions in sequence.

## Set up interactions

To set up an interaction:

1. Select an element on the screen or in the element list.
2. On the right, switch to the **Interactions** panel and click **Add trigger**.

<ZoomImage id="add-trigger.webp" width="500px" alt="Add trigger button" />

3. In the **Button triggers** section, select the [trigger type](#trigger-types).

<ZoomImage id="trigger-type.webp" width="500px" alt="Trigger type selection" />

4. Click **Add action**, click the action name, and select the [action type](#action-types) from the dropdown in the **Edit action** window.

<ZoomImage id="add-action.gif" width="500px" alt="Add action button" />

5. Set up the action [depending on its type](#action-types).
6. If needed, click **Add action** to add more actions for the same trigger.

## Trigger types

Currently, you can add the following triggers to your flow:
- **On tap**: Starts an action when a user taps an element. The most common trigger, since you will need it for navigation between screens.
- **On appear**: Starts an action as soon as an element becomes visible.
- **On submit**: Only for inputs. Mostly used for saving user data as variables.

## Action types

### Navigate to screen

This is the primary action for moving users between screens. It transitions the user from their current view to a new destination screen.

For this action, you only need to set up the destination screen. If you want to enable dynamic navigation, see the [Conditional actions](#conditional-actions) section.

:::important
It should always be the last action in your list. Any action placed after it (like "Set Variable") might not execute because the app has already changed screens.
:::

<ZoomImage id="navigate-to.webp" width="500px" alt="Navigate to screen" />

### Navigate back

Simply returns the user to the previous screen they have seen. This action relies on the actual user journey rather than on the whole screen sequence.

### Open URL

Opens a specific web address. Use this to send users to web pages, articles, or social media profiles outside of your app's native screens.

For this action, you can configure two settings:
- **URL address**: Set a URL address. Additionally, you can make it dynamic — for example, to navigate users to different pages based on their quiz answer or using the data they've submitted. To do this, click <Inline id="variable.webp" alt="Variable icon" /> and select a variable you want to use.
- **Open in external browser**: Define where you want to open external links. By default, they open in an in-app browser to avoid navigating users outside the app. Select the **Open in external browser** checkbox if you want to open links in an external browser.

<ZoomImage id="open-url.webp" width="500px" alt="Open URL" />

### Close flow

Instantly closes the current flow.

### Show/hide elements

Show or hides a specific element on the screen.

This action overrides the initial state set in **Visibility** in the **Design** panel. If **Visibility** is set to **Hide**, the **Show** action will make it appear.

### Show alert

Displays a native system pop-up window. Users must tap **Ok** to proceed.

For alerts, you must set up their **Title** and **Message**. In both, you can use variables to make the content dynamic. To do this, click <Inline id="variable.webp" alt="Variable icon" /> and select a variable you want to use.

<ZoomImage id="show-alert.webp" width="500px" alt="Show alert" />

### Set variable

Saves information to the flow's memory. This action allows you to update the value of any [variable](onboarding-variables) in your project. Before adding an action, create variables in the **Variables** panel on the left.

Use it for remembering quiz answers and inputs, so you can customize the flow.

Click **Add variable** and set as many variables and their values as you need.

<ZoomImage id="set-variable.webp" width="500px" alt="Set variable" />

## Conditional actions

This action allows you to customize the flow and split it into different paths.

Some common use cases include:

- You have a quiz on the screen, and you want to navigate users to different screens based on their answers. In this case, add a conditional action to a button.
- You want to offer different products and offers to different user groups. Place them on different screens and set up conditions for a navigation button.
- You want to skip certain steps for users who have already completed a tutorial in a previous app session.

Conditional actions work like a checklist. The app reads the rules from top to bottom:
1. **IF**: The flow checks the primary condition.
    - Is it True? The flow executes the THEN actions immediately and stops.
    - Is it False? The flow skips to the next section.
2. **ELSE IF**: You can add extra checks here (e.g., "If not Premium, is the user on a Trial?").
3. **ELSE** (Fallback): If none of the rules above matched, the flow executes the actions in this final section.

:::warning
- If a rule is added but has no action assigned, matching the condition results in doing nothing.
- If a variable is added but it is missing an operator or value, the condition is considered invalid.
:::

To set up conditional actions, for each rule, set up a variable to check and an action to perform. You can set more than one action per rule.

:::important
The flow executes only one rule — the first one it matches. If you need to execute both **IF** and **ELSE IF** at the same time, add both actions to **IF**.
:::

<ZoomImage id="conditional-action.webp" width="500px" alt="Conditional actions" />

### Set up selectable elements

When elements are selectable, you can use their state in conditions. For example: `IF quiz.photo is selected, THEN navigate to the Photo screen`.

By default, quizzes, products, tabs, and trial toggles already consist of selectable elements. However, in some cases, you might want to make more components selectable amd add them to groups. For example, you can add a **Don't ask me again** checkbox that will operate as an element inside the quiz.

To make an element selectable:

1. In the **Interactions** tab, select **Turn into selectable** element.
2. Select a group and set the element ID.
3. If you want this option to be default — for example, if you are adding one more quiz answer manually, select the **Set as default in group** checkbox to make this option selected by default.

<ZoomImage id="turn-into-selectable.webp" width="500px" />

### Manage groups

You can see all the groups on the screen in the **Screen settings** panel in the **Selectable groups** section.

Click the group ID to change the ID itself or the group type — single choice, multi-choice, or toggle. Here, you can also see how group elements are referred to in conditions.
