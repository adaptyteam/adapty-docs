The Refund events chart shows how many purchases and subscriptions were refunded. Adapty ties each refund event to the date the refund was issued, not to the subscription start date.

## Calculation

Adapty counts every purchase or subscription refunded within the selected period. Each refund is attributed to the date it happened, not to when the subscription began. Refunds for trials are excluded because trials generate no revenue.

## How metrics handle refunds

Different metrics treat refunds differently. The same refund event may decrease one chart immediately, decrease another chart retroactively (changing past-period values), or not affect a third at all. The matrix below shows the rules per metric.

| Metric | Refunds applied? | Attribution date | Can go negative? | Notes |
| --- | --- | --- | --- | --- |
| [Revenue](revenue) | Yes | Refund date — not the original purchase date | Yes — on days when refunds exceed new revenue | Revenue = total transactions − refunds. |
| [MRR](mrr) | Yes, retroactively | Subscription is removed from all periods it was active | No | Past-period values can decrease after a refund. |
| [ARR](arr) | Yes, retroactively | Same as MRR | No | Past-period values can decrease after a refund. |
| [ARPU](arpu) | Yes | Refund date | Yes (in refund-heavy periods) | Refunds subtract from the revenue numerator. |
| [ARPPU](arppu) | Yes, numerator only | Refund date | Yes (in refund-heavy periods) | Refunds subtract from the revenue numerator. A refunded user still counts in the paying-user denominator, so heavy refunds push ARPPU down faster than expected. |
| [Active subscriptions](active-subscriptions) | Yes, retroactively | Subscription is removed from count | No | |
| [New subscriptions](reactivated-subscriptions) | **No** | — | No | Count includes later-refunded subscriptions. Compare with [Refund events](refund-events) for net impact. |
| [Refund money](refund-money) / [Refund events](refund-events) | Refunds **are** the data | Refund date | No (always ≥ 0) | |
| [Retention](analytics-retention) | **No** | — | No | Refunded users stay counted on the retention curve. This can make Retention look higher than [Active subscriptions](active-subscriptions) or [Revenue](revenue) for the same cohort. |
| [Cohort revenue](analytics-cohorts) | Yes, cumulatively | Refund date | No (cumulative subtractions don't drive cohort revenue below zero) | Refunds subtract from cohort revenue as they occur. |
| [Paywall metrics](paywall-metrics) / [A/B test metrics](results-and-metrics) (counts) | **No** | — | No | Subscribers, Paying Subscribers, and ARPPU counts on these pages don't deduct refunds. |
| GCS / S3 exports | Refund as its own event row | `event_datetime` = refund timestamp | Net columns can go negative when aggregating | The refund row carries `is_refund = true` (S3/GCS) or event type `subscription_refunded` (webhooks). |

### Negative values

In aggregated views (the Revenue chart, custom analytics on exports), a metric can show a negative value for a specific period or grouping when refunds in that bucket exceed new revenue in the same bucket. This isn't a bug — it's the arithmetic working as designed.

For example: a country had no new purchases on a Tuesday, but a $100 refund for an older purchase was processed that day. The country's revenue for Tuesday will show as −$100.
