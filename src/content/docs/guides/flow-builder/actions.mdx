The Interactions Panel is the command center for  an application's logic. While the Design tab handles the visual appearance of elements, the Interactions tab defines their behavior.

Use this panel to create navigation flows, capture user data, process payments, and set up conditional logic.

<ZoomImage id="flow-builder/flow-builder-triggers-and-actions.webp" width="700px" alt="Interactions panel" />

The logic is built upon a simple parent-child hierarchy: Triggers and Actions.
️ Trigger (The "When"): An event that initiates the logic. It defines when the logic should start running. (On Tap, After Delay, On App Open)
Action (The "What"): The specific task the app performs in response to the trigger. A single trigger can execute multiple actions in a sequence. (Navigate to Screen, Set Variable, Purchase Product)

## Triggers

An element should typically have only one trigger of a specific type (e.g., avoid adding two On Tap triggers to the same button). 

### On Tap

The most common trigger. It starts an action when a user taps or clicks on an element (Button, Card, Link).

### On Appear

Fires automatically as soon as the element or screen becomes visible to the user. No interaction required. Best for: Starting animations, or triggering a timer.

### On Changed

Fires every time the data inside an input field changes. Use this to capture what the user is typing or selecting in real-time. Best for: Saving text input to a Variable, slider movements, toggle switches. For text inputs, this must trigger on every keystroke, not just on blur.

### On Scroll Reached Bottom

Fires when the user scrolls a page or text block all the way to the end. Best for: "Terms of Use" screens (unlocking the "I Agree" button only after reading).

### On Playback Finished

Fires specifically when a video file plays to the very end. Best for: Best for: Video Onboarding (auto-advance to the next screen after the video ends)
### On Submit

This trigger is specifically for Input Fields. It fires when the user presses the primary action key on their mobile keyboard (buttons like "Go", "Join", "Done", or "Enter"). ⚠️ Note for Buttons: If user wants to trigger an action when tapping a visual "Continue" button on the screen, use On Tap instead.Best for: Automatically going to the next step immediately after typing, without needing to reach for a button
### On Draw Complete

activates immediately after the chart's line drawing animation finishes rendering. This trigger is designed to synchronize data visualization with conversion elements, such as displaying a purchase button or a promotional message only after the user has viewed the full trend progression.


## Actions

### Navigate to

This is the primary action for moving users between screens. It transitions the user from their current view to a new destination screen and adds this movement to the app’s navigation history.

It should always be the last action in your list. Any action placed after it (like "Set Variable") might not execute because the app has already changed screens.

#### Pick a destination screen

Select which (Target Screen) screen the user should see next.

If the target screen is deleted, this field will show a warning (🔴 Target screen not found).

#### Visual destination editor

Adding this action creates a visible blue connection line (wire) on the canvas from the trigger element to the target screen.

### Open URL

Opens a specific web address. Use this to send users to web pages, articles, or social media profiles outside of your app's native screens.

#### Target browser

📱 In-App Browser (Default)

Behavior: Opens a browser window (Modal) inside your app. The user sees a "Done" or "Close" button to easily return to the previous screen.
Best for: Terms of Service, Privacy Policy, Help Articles.
Why: Keeps the user within your app flow.

🌍 External System Browser

Behavior: Switches apps to Chrome or Safari. The user effectively leaves your app.
Best for: Social Media links (Instagram, Twitter), deeply complex websites, or downloads.
Why: Allows users to use native browser features (saved passwords, tabs).


#### Dynamically set the URL

The URL field supports both Static Links (hardcoded text) and Dynamic Variables (data from your app). Type Manually: Simply click in the field and type a standard address (e.g., https://google.com). This link will be the same for all users.

Insert Variable (Dynamic): Click the { } icon inside the input field to open the variable list. Select a data point (like user.website or selectedProduct.url).
Result: The variable will appear inside the field as a colored tag.To Remove: Place your cursor after the variable tag and press Backspace (or Delete) on your keyboard. The tag will be removed instantly, allowing you to type a static address instead.
Mixing: You can type part of the URL manually and insert a variable at the end (e.g., https://instagram.com/{user_nickname}).

### Navigate back

Simply returns the user one step back to the previous screen. It acts exactly like the standard "Back" button on your phone.

Custom Navigation Bars: If you hide the system navigation bar and draw your own "Back" arrow (\<) in the top-left corner.
Wizards & Quizzes: User is on Step 3 (Email Input) and wants to go back to Step 2 (Plan Selection) to change their mind.
Drill-down Navigation: Settings List -> Profile Settings. The "Back" button returns to the List.

### Close flow

Instantly closes the current modal experience (e.g., a Paywall, Onboarding, or Wizard) and returns the user to the main app content.

For "Exit" scenarios where the user wants to leave the current context entirely.

Paywall "X" Buttons: When the user declines the offer and wants to go to the app.
"Maybe Later" Links: Text links at the bottom of onboarding.
Success Completion: When a user finishes a flow (e.g., "Success! You are subscribed") and clicks "Continue", use Close Flow to drop the paywall and reveal the Home screen.

### Purchase

Initiates the native payment flow (Apple Pay or Google Pay). This action pauses the app, shows the system payment sheet, and processes the transaction.

#### Settings

Dynamic Variable 
When you have multiple options on the screen (e.g., Monthly vs. Yearly cards).
How to set: Click { } inside the input and select a variable like Selected_Product.
Behavior: The button will automatically purchase whichever plan the user currently has selected on the screen.

Specific Product (Hardcoded)
For simple, single-button screens (e.g., a "Buy Lifetime Access" button).
How to set: Select a specific Product ID from your catalog (e.g., premium_annual_1999) directly in the dropdown.
Behavior: This button will always buy this specific item, regardless of other selections.


#### Handle failures and cancellations

Triggered when the transaction fails due to technical reasons (network error, declined card):

Behavior: Enter a custom Title and Message for the alert.
System Fallback: If you leave these fields empty, the app will display the localized default system error message.

Triggered when the user manually closes the payment sheet without paying:

Behavior: Enter a custom Title and Message if you want to show an alert (e.g., "Don't miss out!").
System Fallback: If you leave these fields empty, the app will show the default system message (or nothing, depending on OS version).

#### Web payment options

Configures the button to bypass the App Store and redirect users to a web-based checkout (e.g., Stripe, PayPal). This is used for web paywalls or specific alternative billing flows.

Open in: Choose between In-App Browser (keeps user inside the app) or External Browser (opens Safari/Chrome). Button will open the URL configured in Screen Settings.

### Restore purchases

Checks the user's App Store or Google Play receipt history to reactivate previously purchased content. This is required for users who reinstall the app or switch to a new phone. Apple and Google Guidelines require a "Restore Purchases" button on every screen where a purchase can be made (Paywalls). Failure to include this will result in App Review Rejection.

### Show or hide element

### Change state

This action allows you to switch an element between predefined visual states, such as Active, Disabled, or any custom state you’ve created. When triggered, the element instantly adopts the specific styling and layout configured for that chosen state.

Use Case: Use it to highlight a "Product" container by switching it to a Selected state upon being tapped, or to dim a button by changing its state to Disabled once a purchase process begins.

### Show alert

Displays a native system pop-up window (modal dialog) to the user. This action pauses the user's interaction with the app until they acknowledge the message by tapping "OK" (or "Close").


Errors: "Network Error", "Invalid Email".
Success Messages: "Profile Updated", "Link Copied".
Information: "This feature will be available soon."

### Wait

This action creates a pause in the execution flow before the next action is triggered.

Use Case: Use it to show a "Close" button on a paywall after 3 seconds or to add a small delay before a screen transition to prevent flickering.

### Hide element

Purpose: Hides a specific element from the user interface.

Element: A dropdown list to select the element you want to hide from this screen.

Note: Use this to hide (not delete) items. Conflict Resolution: This action overrides the initial state set in the Visibility property. If an element was hidden by default, this action will make it appear.

### Show element

Toggles element visibility

### Set variable

Saves information to the app's memory. This action allows you to update the value of any variable in your project. It is essential for remembering user choices, inputs, or statuses across different screens.

#### Set value

* Static
* Dynamic

#### Set multiple variables

You can update multiple variables in a single action block.

Click the + Add variable button at the bottom of the panel.

Example: When a user logs in, you can simultaneously set:
is_logged_in = true
login_date = current_time

## Conditional actions

<ZoomImage id="flow-builder/flow-builder-conditional-actions.webp" width="700px" alt="Conditional action settings" />

This action allows users app to make decisions based on user data. It splits the flow into different paths, checking rules from top to bottom until it finds a match.

Examples: checking if a user is a Premium subscriber, filtering users by age, or offering discounts to specific segments.

The Conditional Action works like a checklist. The app reads the rules from top to bottom:
🟣 IF: The app checks the primary condition.
Is it True? The app executes the THEN actions immediately and stops.
Is it False? The app skips to the next section.
🟡 ELSE IF: You can add extra checks here (e.g., "If not Premium, is the user on a Trial?").
🟠 ELSE (Fallback): If none of the rules above matched, the app executes the actions in this final section.

ONLY ONE CONDITIONAL ACTION PER ITEM

💡 Key Rule: The app executes only one branch—the first one that matches.

To keep the interface clean, actions inside the Conditional block are configured in separate layers.

Edit Settings: Click on any action inside a Then an action overlay should be open near the condition overlay

### Troubleshooting conditional actions

The system automatically manages empty or incomplete states to ensure flow stability.

1. Empty Action Containers  If a Then or Else block is left empty (containing no actions), the system defaults to a Do nothing  state. At runtime, the flow simply bypasses this block and proceeds to the next logical step without error.

2. Blank Condition Cleanup Condition rows that are added but left entirely blank are considered temporary. To prevent clutter, these rows are automatically removed from the logic tree when the configuration panel is dismissed or saved.

3. Incomplete Condition Logic A condition is considered "Incomplete" if it has a defined Variable but is missing an Operator or Value (or vice versa).
UI Behavior: The system flags the row with a warning icon (⚠️) to indicate a validation error.

## Custom actions

This action allows you to trigger unique external logic or events by assigning a specific Action ID. It serves as a bridge between the paywall and your app's custom code. Use Case: Use it to fire a specific analytics event with a custom identifier or to trigger a unique native function, such as opening a specialized support chat or a custom mini-game.

## Action types

When configuring actions, choose one of the following types:

#### Navigate
Moves the user to another onboarding screen, letting you control flow based on user actions or selections. Ideal for chaining multiple actions for multi-step logic with quizzes.

#### Show/Hide element
Toggles the visibility of a specified element for conditional content within a screen. Use this to display extra content only when users need it.

#### Open paywall
Launches your app's paywall to present purchases or subscriptions. Learn how to handle opening paywall on [iOS](ios-handling-onboarding-events.md#opening-a-paywall), [Android](android-handle-onboarding-events.md#opening-a-paywall), [React Native](react-native-handling-onboarding-events.md#opening-a-paywall), and [Flutter](flutter-handling-onboarding-events.md#opening-a-paywall).

#### Scroll to
Programmatically scrolls the view to a target element on the current screen. Helpful for long-form screens when a "See details" button is pressed.

#### Custom
Allows you to define and execute your own logic based on the [action ID](#action-id). Use this action to trigger behaviors not covered by the standard action types.

Learn how to handle custom action on [iOS](ios-handling-onboarding-events.md#custom-actions), [Android](android-handle-onboarding-events.md#custom-actions), [React Native](react-native-handling-onboarding-events.md#handle-custom-actions), and [Flutter](flutter-handling-onboarding-events.md#handle-custom-actions).

#### Close onboarding
Ends the onboarding flow and closes the interface. Use when users finish setup to immediately drop back into the main app.

Learn how to handle onboarding closure on [iOS](ios-handling-onboarding-events.md#closing-onboarding), [Android](android-handle-onboarding-events.md#closing-onboarding), [React Native](react-native-handling-onboarding-events.md#closing-onboarding), and [Flutter](flutter-handling-onboarding-events.md#closing-onboarding).

### Action triggers

Actions fire depending on the element they're attached to:

- **Button**: Runs when a user clicks a button or when a timer completes.
- **Quiz**: Executes when an option is selected.
- **Loader**: Triggers after a Loader or Processing finishes.

### Action ID

:::important
Action ID is not the same as the [element ID](onboarding-variables.md) used for inserting dynamic data with variables. Be sure not to mix them up.
:::

When setting up custom actions for buttons, you may want to handle different buttons the same way using action IDs:

1. When [adding a button](onboarding-buttons.md#add-buttons), assign it an ID in the **On Press** section of the **Element** tab.
2. [Use the assigned action ID in your source code](ios-handling-onboarding-events.md#custom-actions).

::::note
On iOS devices, onboardings support only actions in the **On Press** section. The **On Press Extra** section won't work because only one view can be displayed at a time—if one action opens a view (like a paywall), the other action cannot execute simultaneously.
:::
