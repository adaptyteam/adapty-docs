これは Adapty Analytics で利用できる指標の概要です。各指標が何を測定するのか、また関連する指標との違いを理解するためにご活用ください。

Adapty がアナリティクスデータをどのように処理するかについての詳しい説明は、[Adapty アナリティクスの仕組み](how-adapty-analytics-works)をご覧ください。
:::note
この記事では[Adapty アトリビューション](adapty-user-acquisition)の指標については説明していません。広告キャンペーンの指標（費用、CPI、ROAS、CTRなど）について詳しくは、[Adapty アトリビューション アナリティクス](ua-analytics)をご覧ください。
:::
## グローバル指標 \{#global-metrics\}

グローバル指標は、すべてのプレースメントとペイウォールにわたるアプリ全体のパフォーマンスを追跡します。

### 収益 \{#revenue\}

これらの指標は、アプリがどれだけの収益を生み出しているか、またその収益源を測定します。
| 指標 | 説明 | 主な違い |
|--------|-------------|----------------|
| [Revenue](revenue) | サブスクリプションおよび買い切り購入からの総収益（返金分を除く） | 実際に発生した収益。[チャートコントロール](controls-filters-grouping-compare-proceeds)の設定に応じて、グロス収益・手数料控除後の収益・税金と手数料の控除後の収益を表示できます |
| [MRR](mrr) | アクティブなサブスクリプションからの月間経常収益 | アプリの安定した月間収益。買い切り購入および非継続型サブスクリプションは除外されます |
| [ARR](arr) | アクティブなサブスクリプションからの年間経常収益 | MRRと同じ計算方法を年単位で適用したもの。年間収益の予測に役立ちます |
| [ARPU](arpu) | ユーザー1人あたりの平均収益 | 収益を、課金・非課金を問わない全ユーザー数で割った値。各ユーザーが平均的にどれだけの収益をもたらしているかを示します |
| [ARPPU](arppu) | 課金ユーザー1人あたりの平均収益 | 選択期間中に購入を行ったユーザー（返金済みのトランザクションを含む）のみをカウント。常にARPUより高くなります |
| [LTV（ライフタイムバリュー）](ltv) | コホート内の課金顧客からの収益を課金顧客数で割った値 | 課金顧客1人あたりの実現済み価値を時系列で示します。ARPPU（単一期間）とは異なり、LTVは顧客との関係全体にわたる総収益を表します。更新回数別または暦日別で表示可能です |
| [予測LTV](predicted-ltv-and-revenue) | コホートのユーザー1人あたりの推定ライフタイムバリュー | 将来予測の指標。実現済みLTVとは異なり、コホートの過去の継続率パターンから将来の価値を予測します。3・6・9・12・18・24ヶ月で表示可能です |
| [予測収益](predicted-ltv-and-revenue) | コホートが生み出すと推定される総収益 | 将来予測の指標。実現済みRevenueとは異なり、選択した期間にコホートが生み出す総額を予測します。毎日更新されます |
| [Non-subscriptions](non-subscriptions) | アプリ内課金（消耗型アイテム・非消耗型アイテム・非継続型サブスクリプション）の件数 | 自動更新型サブスクリプションは除外されます |
| [Refund events](refund-events) | 返金された購入またはサブスクリプションの件数 | 元の購入日ではなく、返金日に帰属します |
| [Refund money](refund-money) | 選択期間中の返金総額 | 返金による財務的な影響。ストア手数料控除前の金額で算出されます。Refund events（件数）とは異なり、金額を表示します |
### サブスクライバーとコンバージョン \{#subscribers-and-conversion\}

これらの指標は、ユーザーがアプリに流入し、ファネルを通過していく過程を追跡します。
| 指標 | 説明 | 主な違い |
|--------|-------------|----------------|
| [インストール数](installs) | 期間中のアプリインストール数 | [インストールの定義](general#4-installs-definition-for-analytics)に応じて、以下のいずれかをカウント: <br /> • デバイスインストール数（アプリを再インストールしたユーザーは再カウント） <br /> • ユニークユーザー数（`customer_user_id` を設定したユーザーのみカウント。匿名ユーザーは完全に除外 — ユーザーが識別されていない場合、カウントは0） |
| [新規トライアル](new-trials) | 期間中に開始されたトライアル | チャートを表示した時点でトライアルが既に期限切れまたは有料転換済みであっても、すべてのトライアル開始をカウント |
| [アクティブトライアル](active-trials) | まだ期限切れになっていないトライアル数 | 期間終了時点でアクティブなトライアルのみカウント |
| [新規サブスクリプション](reactivated-subscriptions) | 期間中に初めて有効化されたサブスクリプション（トライアルなしの初回購入とトライアルから有料への転換の両方を含む） | 更新や再アクティブ化は除外。トライアル転換は `trial_converted` を発火するため、トライアルなしの初回購入のみをカウントする `subscription_started` インテグレーションイベントとは異なる |
| [アクティブサブスクリプション](active-subscriptions) | まだ期限切れになっていない有料サブスクリプション数 | トライアルと更新がキャンセルされたサブスクリプションは除外 |
| [インストールからトライアルへ](analytics-conversion#install---trial) | トライアルを開始したインストール者の割合 | 分母にはペイウォール閲覧者だけでなくすべてのインストール者が含まれるため、ペイウォール閲覧からトライアルへの割合よりも低くなる場合がある。また、アプリがペイウォール表示をログに記録しない場合、両指標に乖離が生じることがある。これは、`logShowFlow`（iOS SDK v4+）/ `logShowPaywall` を呼び出さないカスタムペイウォールを使用している場合や、ユーザーが[プロモートされたアプリ内課金](https://developer.apple.com/documentation/storekit/supporting-promoted-in-app-purchases-in-your-app)からトライアルを開始した場合に発生する可能性がある。 |
| [ペイウォール閲覧からトライアルへ](analytics-conversion#paywall-view---trial) | トライアルを開始したペイウォール閲覧者の割合 | ペイウォールを閲覧したユーザーのみカウントするため、インストールからトライアルへの割合より高くなる場合がある |
| [トライアルから有料へ](analytics-conversion#trial---paid) | サブスクリプションを購入したトライアルユーザーの割合 | トライアルの質と転換効率を測定。インストールから有料へとは異なり、トライアルを完了したユーザーのみに注目 |
| [インストールから有料へ](analytics-conversion#install---paid) | 最初のサブスクリプションを購入したインストール者の割合 | ペイウォール閲覧者だけでなくすべてのインストール者をカウント。ペイウォール閲覧から有料への割合よりも低くなる場合がある。直接購入とトライアルから有料への転換の両方を含む |
| [ペイウォール閲覧から有料へ](analytics-conversion#paywall-view---paid) | 最終的にサブスクリプションを購入したペイウォール閲覧者の割合 | ペイウォールを閲覧したユーザーのみカウントするため、インストールから有料への割合より高くなる場合がある。トライアルを先に完了したユーザーも含む |
### 維持率とサブスクリプション更新 \{#retention-and-subscription-renewal\}

これらの指標は、アプリが有料サブスクライバーをどの程度長く維持できているかを追跡します。
| 指標 | 説明 | 主な違い |
|--------|-------------|----------------|
| [リテンション](analytics-retention) | 各請求期間（1回目の更新、2回目の更新など）後に残っている元のサブスクライバーの割合 | 初回支払い以降のサブスクライバーを追跡します。以下の期間対期間の指標とは異なり、常に元のグループと比較するため、全体像を一目で把握できます |
| [Paid to 2nd period](analytics-conversion#paid---2nd-period) | 初回サブスクライバーのうち、2回目の期間に更新した割合 | 隣接する2つの特定の期間間のトランジションを測定します。リテンションとは異なり、最も重要な最初の更新に焦点を当てています |
| [2nd to 3rd period](analytics-conversion#2nd-period---3rd-period) | 2回目から3回目の期間に更新した割合 | 初回更新後の初期リテンションの安定性を示します |
| [3rd to 4th period](analytics-conversion#3rd-period---4th-period) | 3回目から4回目の期間に更新した割合 | 中期リテンションの指標 |
| [4th to 5th period](analytics-conversion#4th-period---5th-period) | 4回目から5回目の期間に更新した割合 | 長期ロイヤルティの指標 |
| [6 Months+](analytics-conversion#6-months-) | 初回サブスクライバーのうち、6ヶ月以上サブスクリプションを継続している割合 | 更新回数ではなくカレンダー上の時間を測定します。年間サブスクライバーは、更新がなくても6ヶ月時点でリテンション済みとしてカウントされます |
| [1 Year+](analytics-conversion#1-year-) | 初回サブスクライバーのうち、12ヶ月以上サブスクリプションを継続している割合 | 年間リテンションのマイルストーン |
| [2 Years+](analytics-conversion#2-years-) | 初回サブスクライバーのうち、24ヶ月以上サブスクリプションを継続している割合 | 長期リテンションのマイルストーン |
### チャーン \{#churn\}

これらの指標は、アプリが失ったサブスクリプションユーザーとトライアルユーザーの数を測定します。
| 指標 | 説明 | 主な違い |
|--------|-------------|----------------|
| [トライアル更新キャンセル](trials-renewal-cancelled) | 自動更新を無効にしたユーザーのトライアル | ユーザーはトライアル終了まで引き続きアクセスできるが、自動的に有料へは移行しない。サブスクリプション更新キャンセルとは異なり、まだ支払いをしていないトライアルユーザーに適用される |
| [期限切れ（解約）トライアル](expired-churned-trials) | 期限が切れたトライアル — ユーザーがプレミアム機能へのアクセスを失った | ユーザーはすでにアクセスを失っている。前の期間に更新をキャンセルしていた場合でも、有効期限日に帰属される。理由別（自発的 vs. 請求上の問題）にグループ化可能 |
| [サブスクリプション更新キャンセル](cancelled-subscriptions) | 自動更新を無効にしたユーザーのサブスクリプション | ユーザーは期間終了まで引き続きアクセスできる。実際の解約ではなく解約リスクのシグナル — 期間が終わる前に自動更新を再有効化する可能性がある |
| [解約（期限切れ）サブスクリプション](churned-expired-subscriptions) | 期限が切れたサブスクリプション — ユーザーがプレミアム機能へのアクセスを失った | 実際の解約。ユーザーはすでにアクセスを失っている。前の期間に更新をキャンセルしていた場合でも、有効期限日に帰属される。理由別（自発的 vs. 請求上の問題）にグループ化可能 |
### 請求の問題と収益の回復 \{#billing-issues-and-revenue-recovery\}

これらの指標は、請求の問題によって失われた収益をアプリがどれだけ効果的に回復しているかを追跡します。
| 指標 | 説明 | 主な違い |
|--------|-------------|----------------|
| [グレース期間](grace-period) | 請求失敗によりグレース期間に入ったサブスクリプション | グレース期間を超過してアクセスを失ったユーザーも含む |
| [グレース期間から有料転換](analytics-conversion#grace-period---paid) | グレース期間終了前に更新したグレース期間ユーザーの割合 | 割合（%）。「グレース期間ユーザーのうち何割が回復したか」を示す |
| [グレース期間転換数](grace-period-converted) | 正常に更新されたグレース期間サブスクリプションの絶対数 | グレース期間から有料転換と同じイベントだが、割合ではなく件数で表示 |
| [グレース期間転換収益](grace-period-converted-revenue) | グレース期間回復による収益 | グレース期間機能の財務的インパクト |
| [請求問題](billing-issue) | 請求問題の状態に入ったサブスクリプション | グレース期間終了後に開始。グレース期間とは異なり、すでにプレミアムアクセスを失ったユーザーのみをカウント |
| [請求問題から有料転換](analytics-conversion#billing-issue---paid) | 請求サイクル終了前に更新した請求問題ユーザーの割合 | 割合（%）。「請求問題ユーザーのうち何割が回復したか」を示す |
| [請求問題転換数](billing-issue-converted) | 正常に更新された請求問題サブスクリプションの絶対数 | 正常に更新された請求問題サブスクリプションの件数。請求問題から有料転換と同じイベントだが、割合ではなく件数で表示 |
| [請求問題転換収益](billing-issue-converted-revenue) | 請求問題回復による収益 | 請求問題回復の財務的インパクト |
## ペイウォール、プレースメント、オンボーディングの指標 \{#paywall-placement-and-onboarding-metrics\}

これらの指標は、個々の[ペイウォール](paywall-metrics)、[プレースメント](placement-metrics)、およびオンボーディングに対して計算されます。アプリ全体のパフォーマンスではなく、特定のペイウォールやプレースメントのパフォーマンスを測定するものです。**Associated global metric** 列は、グローバルアナリティクスセクションの対応する指標を示しています。
| 指標 | 説明 | 主な違い | 関連するグローバル指標 |
|--------|-------------|----------------|---------------|
| [収益（Proceeds）](paywall-metrics#proceeds) | 個別プレースメントの税・手数料控除後の収益 | 税・手数料控除後の[Revenue](revenue)に相当 | [Revenue](revenue) |
| [ARPPU](paywall-metrics#arppu) | このペイウォールまたはプレースメントの課金ユーザー1人あたりの平均収益 | グローバルARPPUと同じ計算式だが、単一のペイウォールまたはプレースメントにスコープされる | [ARPPU](arppu) |
| [ARPAS](paywall-metrics#arpas) | 収益をアクティブなサブスクライバー数（トライアルおよび有料）で割った値 | トライアルユーザーを含む。ARPPUと異なり、サブスクライバーベース全体の収益ポテンシャルを反映する | — |
| [ビュー数（Views）](paywall-metrics#views) | ペイウォールまたはプレースメントが表示された合計回数 | 表示ごとにカウント。同じユーザーが同じペイウォールを2回見た場合は2ビューとしてカウントされる | — |
| [ユニークビュー数（Unique views）](paywall-metrics#unique-views) | ペイウォールまたはプレースメントを閲覧したユニークユーザー数 | 何回見ても1ユーザーとして1回カウント。Viewsと異なり、エンゲージメント頻度ではなくリーチを測定する | — |
| [購入へのCR（CR to purchases）](paywall-metrics#cr-to-purchases) | 購入数を総ビュー数で割った値 | 分母として総ビュー数（同じユーザーの繰り返し閲覧を含む）を使用 | [Paywall view to paid](analytics-conversion#paywall-view---paid) |
| [購入へのユニークCR（Unique CR to purchases）](paywall-metrics#unique-conversion-rate-cr-to-purchases) | 購入数をユニークビュー数で割った値 | 分母としてユニークビュー数を使用。繰り返し閲覧ユーザーが1回としてカウントされるため、非ユニークCRより高い率になる | [Paywall view to paid](analytics-conversion#paywall-view---paid) |
| [トライアルへのCR（CR to trials）](paywall-metrics#unique-cr-to-trials) | 開始されたトライアル数を総ビュー数で割った値 | ペイウォールがビューをトライアルにどれだけ効果的にコンバートするかを測定する | [Paywall view to trial](analytics-conversion#paywall-view---trial) |
| [トライアルへのユニークCR（Unique CR to trials）](paywall-metrics#unique-cr-to-trials) | 開始されたトライアル数をユニークビュー数で割った値 | トライアルへのCRと同様の計算だが、分母にユニーク閲覧者数を使用 | [Paywall view to trial](analytics-conversion#paywall-view---trial) |
| [購入数（Purchases）](paywall-metrics#purchases) | このペイウォールにおけるトランザクションの合計数：新規購入、トライアルコンバージョン、アップグレード、ダウングレード、および復帰サブスクリプション | 更新（リニューアル）を除く | [Revenue](revenue) |
| [トライアル数（Trials）](paywall-metrics#trials) | このペイウォールを通じて有効化されたトライアルの合計 | このペイウォールのみにスコープされる | [New trials](new-trials) |
| [トライアルキャンセル数（Trials canceled）](paywall-metrics#trials-canceled) | 自動更新を無効にしたトライアルの数 | このペイウォールのトライアルのみにスコープされる | [Trials renewal cancelled](trials-renewal-cancelled) |
| [返金率（Refund rate）](paywall-metrics#refund-rate) | 返金数を初回購入数（更新を除く）で割った値 | 件数ではなく率（%）。返金数を購入数で正規化する | [Refund events](refund-events)（率ではなく件数） |
| 完了数（Completions） | ユーザーがオンボーディングフローを最初から最後の画面まで完了した回数 | プレースメントおよびオンボーディングのみ。同じユーザーによる繰り返し完了を含む、すべての完了をカウントする | — |
| ユニーク完了数（Unique completions） | オンボーディングフローを完了したユニークユーザー数 | プレースメントおよびオンボーディングのみ。各ユーザーは1回としてカウント。Completionsと異なり、フローを完了した個人の数を測定する | — |
| ユニーク完了率（Unique completions rate） | ユニーク完了数をユニークビュー数で割った値 | プレースメントおよびオンボーディングのみ。オンボーディングの効果を測定：開始したユーザーのうち実際に完了した割合 | — |
