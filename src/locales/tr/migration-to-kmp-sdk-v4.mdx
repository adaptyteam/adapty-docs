Adapty Kotlin Multiplatform SDK 4.0 (beta), flow'ları tanıtıyor ve paywall API'lerini buna göre yeniden adlandırıyor. Yeni API'ler hem yeni Flow Builder hem de mevcut Paywall Builder ile çalışır — Adapty Kontrol Paneli tarafında herhangi bir kurulum değişikliğine gerek yoktur.

## Hızlı başvuru \{#quick-reference\}

| v3 | v4 |
|---|---|
| `Adapty.getPaywall(placementId, locale)` | `Adapty.getFlow(placementId)` |
| `Adapty.getPaywallForDefaultAudience(placementId, locale)` | `Adapty.getFlowForDefaultAudience(placementId)` |
| `Adapty.getPaywallProducts(paywall)` | `Adapty.getPaywallProducts(flow)` |
| `Adapty.logShowPaywall(paywall)` | `Adapty.logShowFlow(flow)` |
| `AdaptyPaywall` | `AdaptyFlow` |
| `AdaptyUI.createPaywallView(paywall, ...)` | `AdaptyUI.createFlowView(flow, ...)` |
| `AdaptyUI.createNativePaywallView(...)` → `AdaptyNativePaywallView` | `AdaptyUI.createNativeFlowView(...)` → `AdaptyNativeFlowView` |
| `AdaptyUIPaywallView` | `AdaptyUIFlowView` |
| `AdaptyUI.presentPaywallView(view)` / `dismissPaywallView(view)` | `AdaptyUI.presentFlowView(view)` / `dismissFlowView(view)` |
| `AdaptyUI.setPaywallsEventsObserver(observer)` | `AdaptyUI.setFlowsEventsObserver(observer)` |
| `AdaptyUI.registerPaywallEventsListener` / `unregisterPaywallEventsListener` | `AdaptyUI.registerFlowEventsListener` / `unregisterFlowEventsListener` |
| `AdaptyUIPaywallsEventsObserver` | `AdaptyUIFlowsEventsObserver` |
| `AdaptyUIPaywallPlatformView(paywall, ...)` | `AdaptyUIFlowPlatformView(flow, ...)` |
| `paywallViewDidPerformAction`, `paywallViewDidAppear` ve diğer `paywallView...` callback'leri | `flowViewDidPerformAction`, `flowViewDidAppear` ve diğer `flowView...` callback'leri |
| `paywallViewDidFailRendering` | `flowViewDidReceiveError` |

`AdaptyPaywallProduct` adını korur — ürünler hâlâ bir flow'a aittir ve `getPaywallProducts` de adını korur, artık bir `AdaptyFlow` alır. `getFlow` ve `getFlowForDefaultAudience` metodları artık `locale` parametresi almaz. Satın alma ve profil API'leri (`makePurchase`, `restorePurchases`, `getProfile`, `identify`, `updateProfile`) ve `setFallback` aracılığıyla yedekler değişmeden kaldı. Onboarding metodları hâlâ çalışıyor ancak kullanımdan kaldırıldı — bkz. [Onboarding API'sinin kullanımdan kaldırılması](#onboarding-api-deprecation). Bazı varsayılan davranışlar değişti — bkz. [Varsayılan davranış değişiklikleri](#default-behavior-changes).

## Kurulum \{#installation\}

v4.0 bir ön sürüm olduğundan, tam sürümü sabitleyin — Gradle, dinamik aralıklar aracılığıyla ön sürümleri seçmez:

```toml showLineNumbers title="libs.versions.toml"
[versions]
adapty-kmp = "4.0.0-beta.1"

[libraries]
adapty-kmp = { module = "io.adapty:adapty-kmp", version.ref = "adapty-kmp" }
adapty-kmp-ui = { module = "io.adapty:adapty-kmp-ui", version.ref = "adapty-kmp" }
```

`adapty-kmp-ui` modülü yalnızca flow'ları ve paywallları Compose Multiplatform katmanı (`view.present()`) ile render etmek istediğinizde gereklidir. Tam kurulum için bkz. [Adapty SDK Kurulumu](sdk-installation-kotlin-multiplatform).

Her iki platformda yerel Adapty SDK'ları 4.x sürümlerine güncellendi ve otomatik olarak çözümleniyor — herhangi bir derleme değişikliği yapmanıza gerek yok. iOS deployment target **15.0** olarak kalıyor, bu sürümde değişmedi.

## Flow'ları Getirme \{#fetching-flows\}

### getPaywall → getFlow

Döndürülen tür `AdaptyPaywall`'dan `AdaptyFlow`'a değişir ve `locale` parametresi kaldırılır — bir flow render ettiğinizde locale otomatik olarak çözümlenir; özel paywall'lar için tüm locale'ler `flow.remoteConfigs` içinde döndürülür:

```diff showLineNumbers
- Adapty.getPaywall("YOUR_PLACEMENT_ID", locale = "en")
-     .onSuccess { paywall ->
-         // use the paywall
+ Adapty.getFlow("YOUR_PLACEMENT_ID")
+     .onSuccess { flow ->
+         // use the flow
      }
      .onError { error ->
          // handle the error
      }
```

`getPaywallForDefaultAudience` aynı şekilde yeniden adlandırılır:

```diff showLineNumbers
- Adapty.getPaywallForDefaultAudience("YOUR_PLACEMENT_ID", locale = "en")
+ Adapty.getFlowForDefaultAudience("YOUR_PLACEMENT_ID")
```

### getPaywallProducts(paywall) → getPaywallProducts(flow)

`getPaywallProducts` adını korur ancak artık bir `AdaptyFlow` alır:

```diff showLineNumbers
- Adapty.getPaywallProducts(paywall)
+ Adapty.getPaywallProducts(flow)
      .onSuccess { products ->
          // use the products
      }
```

## Veri modeli \{#data-model\}

`getFlow`, `AdaptyPaywall` yerine bir `AdaptyFlow` döndürür ve nesne yapısı değişti:

| v3 `AdaptyPaywall` özelliği | v4 `AdaptyFlow` özelliği | İşlem |
|---|---|---|
| `remoteConfig: AdaptyRemoteConfig?` (tekil) | `remoteConfigs: List<AdaptyRemoteConfig>` | Bir flow, yapılandırılmış her dil için ayrı bir remote config taşır. Kullanıcıya uygun olanı okuyun: `flow.remoteConfigs.firstOrNull { it.locale == "en" }`. |
| _(yeni)_ | `paywalls: List<AdaptyFlowPaywall>` | Her giriş, flow içindeki bir paywall varyasyonunu temsil eder; kendi `name`, `variationId` ve `productIdentifiers` alanlarına sahiptir. Web paywall metodları `AdaptyFlowPaywall` alır — bkz. [Web paywall metodları](#web-paywall-methods). |
| `productIdentifiers` | taşındı | Ürün tanımlayıcıları artık her varyasyonda bulunur: `flow.paywalls[i].productIdentifiers`. Ürünleri getirmek için `getPaywallProducts(flow)` çağrısını kullanmaya devam edin. |
| `hasViewConfiguration` | kaldırıldı | Kodunuzdaki tüm `hasViewConfiguration` kontrollerini kaldırın — bunun yerine `createFlowView` bir hata döndürür (bkz. [Flow'ları görüntüleme](#displaying-flows)). |

`hasViewConfiguration`, `AdaptyOnboarding` üzerinde kalmaya devam eder — yalnızca flow modeli bunu kaldırır.

## Web paywall metodları \{#web-paywall-methods\}

`openWebPaywall` ve `createWebPaywallUrl` isimlerini korur, ancak `paywall` parametresi, `flow.paywalls` içindeki varyantlardan biri olan `AdaptyFlowPaywall` alan bir `flowPaywall` parametresiyle değiştirilir. Yine de `AdaptyPaywallProduct` geçirebilirsiniz:

```diff showLineNumbers
- Adapty.openWebPaywall(paywall = paywall)
+ flow.paywalls.firstOrNull()?.let { flowPaywall ->
+     Adapty.openWebPaywall(flowPaywall = flowPaywall)
+ }
```

## Flow görüntülemelerini takip etme \{#tracking-flow-views\}

### logShowPaywall → logShowFlow

`logShowPaywall`, `logShowFlow` olarak yeniden adlandırıldı ve artık bir `AdaptyFlow` alıyor. Etkinlik yine aynı varyasyon üzerinden kaydedilir; bu nedenle mevcut dönüşüm hunisi ve A/B testi metrikleri kontrol panelinde herhangi bir değişiklik yapılmadan çalışmaya devam eder.

```diff showLineNumbers
- Adapty.logShowPaywall(paywall)
+ Adapty.logShowFlow(flow)
```

v3'te olduğu gibi, [Flow Builder](adapty-flow-builder) veya [Paywall Builder](adapty-paywall-builder) tarafından oluşturulan flow'ları ya da paywall'ları görüntülerken bu metodu çağırmanıza gerek yoktur — Adapty bu görüntülemeleri otomatik olarak takip eder.

## Flow'ları görüntüleme \{#displaying-flows\}

### createPaywallView → createFlowView

Factory metodunu yeniden adlandırın ve `AdaptyFlow` nesnesi geçirin. Döndürülen görünüm türü `AdaptyUIPaywallView`'dan `AdaptyUIFlowView`'a yeniden adlandırılmıştır; ancak metodları (`present`, `dismiss`) ve isteğe bağlı parametreler (`loadTimeout`, `preloadProducts`, `customTags`, `customTimers`, `customAssets`, `productPurchaseParams`) değişmeden kalmaktadır:

```diff showLineNumbers
- AdaptyUI.createPaywallView(paywall)
+ AdaptyUI.createFlowView(flow)
      .onSuccess { view ->
          view.present()
      }
      .onError { error ->
          // handle the error
      }
```

Compose Multiplatform kullanmıyorsanız, native factory metodu da aynı şekilde yeniden adlandırılır:

```diff showLineNumbers
- AdaptyUI.createNativePaywallView(paywall)
+ AdaptyUI.createNativeFlowView(flow)
```

`createFlowView`, flow için herhangi bir view yapılandırması yoksa `AdaptyResult.Error` döndürür — bu, v3'teki `hasViewConfiguration` kontrolünün yerini alır:

```diff showLineNumbers
- if (paywall.hasViewConfiguration) {
-     AdaptyUI.createPaywallView(paywall)
-         .onSuccess { view -> view.present() }
- }
+ AdaptyUI.createFlowView(flow)
+     .onSuccess { view -> view.present() }
+     .onError { error ->
+         // the flow has no view configured, or view creation failed
+     }
```

:::note
Flow view tek kullanımlıktır: `dismiss()` çağırdıktan sonra view yok edilir, bu nedenle flow'u tekrar göstermek için `createFlowView` metodunu yeniden çağırmanız gerekir.
:::

## Olayları işleme \{#handling-events\}

Olaylar gözlemcisi `AdaptyUIPaywallsEventsObserver` adından `AdaptyUIFlowsEventsObserver` adına yeniden adlandırıldı ve callback'lerinin `paywallView` öneki `flowView` olarak değiştirildi. Mevcut handler gövdelerinde kod değişikliğine gerek yok — sadece türü ve override'ları yeniden adlandırın:

```diff showLineNumbers
- AdaptyUI.setPaywallsEventsObserver(object : AdaptyUIPaywallsEventsObserver {
-     override fun paywallViewDidFinishPurchase(
-         view: AdaptyUIPaywallView,
+ AdaptyUI.setFlowsEventsObserver(object : AdaptyUIFlowsEventsObserver {
+     override fun flowViewDidFinishPurchase(
+         view: AdaptyUIFlowView,
          product: AdaptyPaywallProduct,
          purchaseResult: AdaptyPurchaseResult
      ) {
          // custom logic after purchase
      }
  })
```

Bir callback de yeniden adlandırıldı: `paywallViewDidFailRendering`, `flowViewDidReceiveError` olarak değiştirildi. Bu callback, öncekiyle aynı render hatalarında, artı diğer satın alma dışı çalışma zamanı hatalarında tetiklenir:

```diff showLineNumbers
- override fun paywallViewDidFailRendering(view: AdaptyUIPaywallView, error: AdaptyError) {}
+ override fun flowViewDidReceiveError(view: AdaptyUIFlowView, error: AdaptyError) {}
```

Callback'lerin tam listesi için [Flow & paywall olaylarını yönetme](kmp-handling-events) bölümüne bakın.

### Compose platform view

Compose Multiplatform composable ile görünümleri gömerseniz, `AdaptyUIPaywallPlatformView(paywall, ...)` bileşeni `AdaptyUIFlowPlatformView(flow, ...)` olarak yeniden adlandırılmıştır. Event callback'leri `onDid...` adlarını korur; yalnızca `onDidFailRendering`, `onDidReceiveError` olarak değişir:

```diff showLineNumbers
- AdaptyUIPaywallPlatformView(
-     paywall = paywall,
+ AdaptyUIFlowPlatformView(
+     flow = flow,
      onDidFinishPurchase = { view, product, result -> /* ... */ },
  )
```

v3'te olduğu gibi, burada ilettiğiniz callback'ler (ve `registerFlowEventsListener` aracılığıyla kaydedilen tüm observer'lar) global observer'ın **yerine değil, onun ek olarak** çalışır — callback'iniz bir olayı gözlemler; global varsayılanın yerini almaz. [Değiştirilen varsayılanları](#default-behavior-changes) aklınızda bulundurun: örneğin, global varsayılan artık satın alma işleminin ardından görünümü kapatmaz.

### Yeni API'ler

- `AdaptyUI.setObserverModeResolver(...)` ve `AdaptyUIObserverModeResolver` — SDK [Observer mode](implement-observer-mode-kmp)'da çalışırken flow'lardan başlatılan satın alma ve geri yükleme işlemlerini yönetin. Daha önce bu özellik yalnızca native iOS ve Android SDK'larında mevcuttu. Bkz. [Flow'ları Observer mode'da sunma](kmp-present-flows-in-observer-mode).
- `AdaptyUI.setSystemRequestsHandler(...)` ve `AdaptyUISystemRequestsHandler` — bir flow'dan gelen sistem istekleri için ayrılmıştır (işletim sistemi izin istemleri ve uygulama değerlendirme istekleri). Flow'lar henüz bu istekleri tetiklemediğinden bir işleyici kaydetmenize gerek yoktur.
- Yeni isteğe bağlı `flowViewDidReceiveAnalyticEvent` callback'i, bir flow'dan gelen özel analitik olaylar için ayrılmıştır. Flow'lar henüz bu olayları kodunuza iletmediğinden bunu uygulamanıza gerek yoktur.
- `AdaptyUI.openWebUrl(url, openIn)` ve `AdaptyUI.requestAppReview()` — bunlar varsayılan `OpenUrlAction` işlemini ve varsayılan `handleAppReviewRequest`'i destekler; böylece URL'ler ve uygulama değerlendirme istemleri kutudan çıktığı gibi native olarak işlenir. Bunları yalnızca varsayılanları geçersiz kılarsanız doğrudan çağırın.
- `AdaptyConfig.ServerCluster.CN` — uygulamanızı [Adapty'nin Çin sunucularına](china-cluster) bağlamak için `DEFAULT` ve `EU`'nun yanına eklenen yeni bir sunucu kümesi seçeneği.

## Varsayılan davranış değişiklikleri \{#default-behavior-changes\}

Bu değişiklikler derleme hatalarına yol açmaz, bu yüzden çalışma zamanında test edin:

- **Satın alma tamamlama**: v3'te varsayılan `paywallViewDidFinishPurchase`, `AdaptyPurchaseResult.UserCanceled` dışındaki herhangi bir satın alma sonucundan sonra görünümü kapatıyordu. v4'te varsayılan `flowViewDidFinishPurchase` hiçbir şey yapmaz; dolayısıyla **bir flow, siz kapatana kadar satın alma sonrasında açık kalır** — bu iOS davranışıyla örtüşür. Otomatik kapanmaya güveniyorsanız, satın alma tamamlanınca `view.dismiss()` çağrısını kendiniz yapın.
- **Android sistem geri tuşu**: v3'te varsayılan `paywallViewDidPerformAction`, hem `CloseAction` hem de `AndroidSystemBackAction` durumunda görünümü kapatıyordu. v4'te varsayılan yalnızca `CloseAction`'ı yönetir — **sistem geri tuşu artık bir flow'u tek başına kapatmaz**; bu, bir flow'un sistem hareketiyle kapatılamadığı iOS davranışıyla örtüşür. Kullanıcılara açık bir çıkış yolu sunun (bir **Kapat** düğmesi veya `on_device_back` eylemi) ya da `flowViewDidPerformAction` içinde görünümü kendiniz kapatın.
- **Görünüm hataları**: v3'te varsayılan `paywallViewDidFailRendering` hiçbir şey yapmıyordu. v4'te varsayılan `flowViewDidReceiveError` **görünümü kapatır** — görünümü açık tutmak ya da hatayı farklı şekilde işlemek istiyorsanız bu davranışı geçersiz kılın.
- **Görünümler tek kullanımlıktır**: `dismiss()` çağrısının ardından görünüm yok edilir. Flow'u yeniden göstermek için `createFlowView`'ı tekrar çağırın.

## Onboarding API'sinin Kullanımdan Kaldırılması \{#onboarding-api-deprecation\}

Eski onboarding API'si, v4.0'da [Flow Builder](adapty-flow-builder) lehine kullanımdan kaldırılmıştır. Hâlâ çalışmaktadır, ancak gelecekteki bir sürümde kaldırılacaktır; bu nedenle onboarding'lerinizi Flow Builder'a taşımayı planlayın.

Kullanımdan kaldırılan semboller: `getOnboarding`, `getOnboardingForDefaultAudience`, `AdaptyUI.createOnboardingView`, `AdaptyUI.createNativeOnboardingView` ve `AdaptyUIOnboardingsEventsObserver`.
