**Interactions** 面板用于定义流程元素如何响应事件——例如点击、元素出现以及表单提交。对于每个事件，你可以分配一个或多个操作：在屏幕之间导航、显示或隐藏元素、打开 URL、设置变量等。使用条件可根据用户数据自定义流程。

每个交互遵循三部分链式结构：
1. **Element（元素）**：触发交互的屏幕组件——按钮、测验答案、输入框等。
2. **Trigger（触发器）**：激活逻辑的事件，例如点击、元素出现或表单提交。
3. **Action（动作）**：流程响应时执行的任务。一个触发器可以按顺序执行多个动作。
## 设置交互 \{#set-up-interactions\}

设置交互的步骤如下：

1. 在画布或 **Layers** 面板中选中一个元素。
2. 在右侧切换到 **Interactions** 面板，点击 **Add trigger**。

<ZoomImage id="add-trigger.webp" width="500px" alt="Add trigger button" />

3. 在 **Button triggers** 区域，选择[触发器类型](#trigger-types)。

<ZoomImage id="trigger-type.webp" width="500px" alt="Trigger type selection" />

4. 点击 **Add action**，点击动作名称，然后在 **Edit action** 窗口的下拉菜单中选择[动作类型](#action-types)。
<ZoomImage id="add-action.gif" width="500px" alt="Add action button" />

5. 根据您选择的[操作类型](#action-types)配置操作属性。
6. 如需添加更多操作，请点击 **Add action**。
## 触发器类型 \{#trigger-types\}

触发器会响应用户行为、元素状态变化或页面加载而触发。**On screen appear** 是通用触发器，其余触发器仅适用于特定元素。
| 触发器 | 触发时机 | 支持的元素 |
|---|---|---|
| **On screen appear** | 屏幕加载时 | 所有元素 |
| **On tap** | 用户点击该元素时 | [按钮](paywall-buttons)、[答题选项](onboarding-quizzes)、[开关](builder-toggles)、[倒计时](flow-timer)、[视频](custom-media) |
| **On changed** | 用户更改输入值时（输入文字、选择日期或时间） | 所有[输入元素](builder-inputs-and-forms) |
| **On submit** | 用户通过按下键盘上的 Enter 或完成键提交文本输入时 | [文本类输入](builder-inputs-and-forms) |
| **On timer end** | [倒计时](flow-timer)元素归零时 | [倒计时](flow-timer) |
| **On playback finished** | [视频](custom-media)播放结束时 | [视频](custom-media) |
对于没有内置交互的元素（例如 [Loader](builder-loaders-and-progress-bars)），**On screen appear** 是唯一可用的触发器。
## 操作类型 \{#action-types\}

:::important
**任何导航操作**（即将用户切换到其他页面的操作）都应始终放在操作列表的最后。排在它之后的操作（例如"设置变量"）可能不会执行，因为应用已经切换页面了。
:::
### 导航至指定屏幕 \{#navigate-to-screen\}

这是在屏幕之间切换用户的主要操作，可将用户带到指定的目标屏幕。

使用此操作时，只需设置目标屏幕即可。如需启用动态导航，请参阅[导航与分支](onboarding-navigation-branching)或[条件操作](#conditional-actions)部分。

<ZoomImage id="navigate-to.webp" width="500px" alt="Navigate to screen" />
### 导航到下一屏 \{#navigate-next\}

将用户推进到流程中的下一个屏幕。适用于线性流程——屏幕在编辑器中的排列顺序即为用户实际看到的顺序。
### 返回上一页 \{#navigate-back\}

返回用户导航历史中的上一个页面，而非序列中的上一个页面。
### 打开 URL \{#open-url\}

:::tip
使用[内联链接](onboarding-text#inline-link)将链接插入到正文文本中。
:::

打开指定的网址。可用于将用户跳转到应用原生页面以外的网页、文章或社交媒体主页。
对于此操作，您可以配置两个设置：
- **URL address**：设置一个 URL 地址。此外，您还可以将其设为动态地址——例如，根据用户的测验答案或其提交的数据将用户导航到不同页面。为此，请点击 <Inline id="variable.webp" alt="Variable icon" /> 并选择您想使用的变量。
- **Open in external browser**：定义外部链接的打开方式。默认情况下，链接会在应用内浏览器中打开，以保持用户留在应用内。如果您希望在外部浏览器中打开链接，请勾选 **Open in external browser** 复选框。
<ZoomImage id="open-url.webp" width="500px" alt="Open URL" />
### 关闭流程 \{#close-flow\}

关闭当前流程。
### 显示/隐藏元素 \{#showhide-elements\}

显示或隐藏屏幕上的特定元素。

此操作会覆盖**设计**面板中**可见性**的初始状态设置。如果**可见性**设置为**隐藏**，则**显示**操作会使其出现。

:::important
没有目标元素的**显示**或**隐藏**操作会[阻止预览和发布](builder-save-publish#troubleshooting)。请选择目标元素或移除该操作。
:::
### 显示提示框 \{#show-alert\}

显示原生系统弹出窗口。用户必须点击 **Ok** 才能继续。

对于提示框，你需要设置其 **Title** 和 **Message**。在这两个字段中，你都可以使用变量来实现动态内容。点击 <Inline id="variable.webp" alt="Variable icon" /> 并选择要使用的变量即可。

:::important
配置为空或不完整的 **Show alert** 操作会[阻止预览和发布](builder-save-publish#troubleshooting)。请填写两个字段，或删除该操作。
:::

<ZoomImage id="show-alert.webp" width="500px" alt="Show alert" />
### 设置变量 \{#set-variable\}

更新流程中某个变量的值。在添加此操作之前，请先在左侧 **Variables** 面板中创建变量（参见[变量](onboarding-variables)）。

点击 **Add variable**，根据需要设置任意数量的变量及其值。

:::important
未配置赋值的 **Set variable** 操作会[阻止预览和发布](builder-save-publish#troubleshooting)。请至少配置一条赋值，或删除该操作。
:::

<ZoomImage id="set-variable.webp" width="500px" alt="Set variable" />
### 购买 \{#purchase\}

通过用户引导中的按钮或交互直接触发购买流程。使用此功能可让用户无需离开流程即可完成订阅或购买产品。
您可以为此操作配置两种行为：
- **In-app store**：发起原生购买。将 **Product** 设置为特定产品，或设置为 `products.selectedProduct` 以使用用户在当前页面的选择。
- **Web payment**：将用户跳转到[网页付费墙](web-paywall)，而非触发原生购买。适用于需要在应用外处理交易的场景，例如基于网页的订阅服务。
:::important
**Purchase**（购买）操作如果没有指定目标**产品**或 **Web Paywall URL**，将会[阻止预览和发布](builder-save-publish#troubleshooting)。请为其分配一个目标，或删除该操作。
:::

<ZoomImage id="onboarding-purchase-action.webp" width="500px" alt="Purchase action configuration in the Edit action window" />
### 恢复购买 \{#restore-purchases\}

在设备上触发恢复购买流程。当用户之前在其他设备上购买过订阅，或重新安装应用后需要恢复其权益时，可点击此按钮。

此操作无需任何配置——Adapty 会通过原生商店流程处理恢复操作。

**Restore purchases** 操作也预先配置在 **Links** 按钮预设的 **Restore** 链接上（请参阅[设置购买](paywall-product-block#restore-purchases)）。
## 自定义动作 \{#custom-actions\}

自定义动作会触发一个具名的 **Action ID**，由你自己的应用代码来处理。当内置动作类型无法满足需求时，可以使用它。

Adapty 负责触发，你的应用负责实现具体行为：

1. 在编辑工具中，为某个元素的交互指定一个 **Action ID**。
2. 当用户触发该交互时，流程会将此 ID 传递给你的应用。
3. 你的应用根据 ID 进行匹配，并执行相应代码。
### 设置自定义操作 \{#set-up-a-custom-action\}

1. 在 **Edit action** 窗口中，分配一个 **Action ID**——这是一个你的应用代码能够识别的字符串（例如 `show_discount`）。
2. 在应用代码中，为该 Action ID 实现对应的处理逻辑。详细说明和代码示例请参阅[处理付费墙操作](handle-paywall-actions)。

:::important
没有 **Action ID** 的 **Custom** 操作会[阻止预览和发布](builder-save-publish#troubleshooting)。请为该操作分配 Action ID 或将其删除。
:::
<ZoomImage id="onboarding-custom-action.webp" width="500px" alt="编辑操作窗口中带有 Action ID 字段的自定义操作配置" />
### 自定义操作的用途 \{#what-you-can-do-with-custom-actions\}

自定义操作本身不会执行任何操作。你在编辑工具中设置一个静态 Action ID，由应用代码来处理接收到该 ID 后的逻辑。以下所有使用场景都遵循同一模式：在流程中分配一个 ID，然后在代码中处理它。
- **触发应用内事件**：触发一个 ID（例如 `viewed_special_offer`），然后在应用收到该事件时将其记录到你的分析系统中。
- **请求系统权限**：触发一个 ID（例如 `request_location`），然后在应用中调用操作系统的权限弹窗。对于无法通过弹窗授权的权限，直接跳转到手机系统设置。Adapty 不会显示弹窗——这由你的应用来完成。
- **启动原生身份验证**：触发一个 ID（例如 `login_google`），然后展示你自己的登录界面。该流程无法直接为用户完成登录操作。
- **执行业务逻辑**：触发一个 ID（例如 `apply_discount`），然后在你这一侧解锁内容或更改应用状态。
- **将测验答案传递给应用**：为每个选项分配不同的 Action ID（例如 `goal_weight_loss` 和 `goal_muscle`），然后在代码中读取该 ID。利用这个 ID 设置一个[自定义用户属性](setting-user-attributes#custom-user-attributes)，以便后续进行市场细分。由于该 Action 只携带固定的 ID，这是上报用户选择的唯一方式——流程本身无法发送所选的值。
:::important
自定义动作在用户选择选项的瞬间触发。如果用户更改了答案，流程会再次触发新的 Action ID。你的应用将按顺序接收到两个动作——例如先收到 `goal_weight_loss`，再收到 `goal_muscle`。请确保你的处理逻辑具有幂等性，以便最新的信号生效。
:::
### 自定义 Action 的局限性 \{#what-custom-actions-cant-do\}

自定义 Action 是静态的。Action ID 在构建流程时就已固定——无法读取[变量](onboarding-variables)或[用户输入](builder-inputs-and-forms)。当 Action 触发时，你的应用只会收到该 ID，而不会获取用户填写的邮箱、手机号或其他任何输入内容。输入字段仅作为变量保留在流程内部，用于分支逻辑和个性化处理。如需在应用中使用这些数据，请通过自己的 UI 或 API 进行采集。
自定义动作同样是单向的。您的应用无法向流程返回结果，流程也不会等待您的代码执行完毕。如果自定义动作之后紧跟一个 **Navigate next** 动作，即使您的代码执行失败，用户也会跳转到下一个屏幕——例如，当用户关闭登录页面而未完成登录时。再加上静态 Action ID 的限制，这意味着您无法在应用中校验用户输入——例如，检查用户输入的短信验证码并根据结果进行分支跳转。如果后续流程依赖于您的代码执行结果，请[将流程拆分到两个版位](#continue-the-flow-based-on-the-result)。
### 根据结果继续流程 \{#continue-the-flow-based-on-the-result\}

如果某些页面只应在自定义操作成功后才显示——例如登录后才出现的页面——可以将流程拆分为两个[版位](placements)，由应用决定何时显示第二部分：

1. 在第一个版位中，创建一个以自定义操作结束的流程（例如 `login`）。
2. 在应用中处理该 Action ID：展示你的登录页面并验证用户是否已登录。
3. 如果用户已登录，则从第二个版位显示包含后续页面的流程。
这样，您的应用就能根据实际结果来控制页面跳转，而不是让流程不论结果如何都自动向前导航。
## 条件操作 \{#conditional-actions\}

使用条件操作可根据用户数据将流程拆分为不同的路径。

常见使用场景包括：

- 屏幕上有问卷，希望根据用户的回答将其导航到不同的页面。此时，可为按钮添加条件操作。
- 希望向不同用户群体展示不同的产品和优惠。将它们放在不同的页面上，并为导航按钮设置相应条件。
- 希望跳过已在上次会话中完成过引导教程的用户的某些步骤。
条件动作的工作方式类似于 if / else-if / else 链。应用从上到下依次读取规则，遇到第一个匹配项即停止：
1. **IF**：流程检查主要条件。
    - 结果为 True？流程立即执行 THEN 动作并停止。
    - 结果为 False？流程跳至下一节。
2. **ELSE IF**：可在此添加额外检查（例如："如果不是 Premium 用户，是否处于试用期？"）。
3. **ELSE**（兜底）：如果以上所有规则均未匹配，流程将执行此最终节中的动作。
:::important
- 如果添加了规则但未分配任何动作，匹配该条件将不执行任何操作。
- 不完整的规则（缺少运算符或值）会[阻止预览和发布](builder-save-publish#troubleshooting)。
:::

对于每条规则，选择要评估的变量和要执行的动作。每条规则可以设置多个动作。

:::important
流程只执行第一条匹配的规则。如果需要同时执行 **IF** 和 **ELSE IF**，请将两个动作都添加到 **IF** 中。
:::
要了解如何使元素可选并将其组织成组以在条件中使用，请参阅[可选元素与组](flow-selectable-elements)。
