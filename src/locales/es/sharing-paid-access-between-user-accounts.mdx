Cuando un usuario realiza una compra, Adapty asigna un nuevo [nivel de acceso](access-level) a su [perfil](identifying-users) activo. Este nivel de acceso autoriza al comprador a acceder al contenido de pago.

El perfil del comprador puede cambiar sin querer si reinstala la app o inicia sesión con una nueva cuenta dentro de la aplicación. Para garantizar un acceso ininterrumpido, Adapty comparte automáticamente el nivel de acceso del usuario entre el perfil original y los que vengan después.

Este enfoque funciona bien para la mayoría de las aplicaciones. Sin embargo, si tu lógica de negocio lo requiere, puedes elegir una política de compartición de acceso de pago más restrictiva.

Abre la página [General Settings](https://app.adapty.io/settings/general) para configurar una política de compartición de nivel de acceso. Para facilitar las pruebas, puedes cambiar esta configuración únicamente para el [entorno sandbox](#sharing-paid-access-on-sandbox).

<Details>

:::important
Si tu aplicación no autentica a los usuarios, puedes ignorar esta configuración. Los perfiles anónimos asociados a la misma cuenta de store *siempre* comparten su nivel de acceso.
:::

## Activado (predeterminado) \{#enabled-default\}

Esta opción es la más adecuada para aplicaciones **sin autenticación integrada**. Tras la compra, todos los perfiles asociados a la misma cuenta de store heredan automáticamente el nivel de acceso.

* Si un usuario inicia sesión en tu app con nuevas credenciales, mantiene el acceso al contenido de pago.
* Si un usuario reinstala la aplicación después de un restablecimiento de fábrica, mantiene el acceso al contenido de pago.
* Si un usuario instala la aplicación en otros dispositivos con la misma cuenta de store, la compra estará disponible en todos ellos, aunque cada instancia de la aplicación tenga su propio perfil de cliente.

## Transferir el acceso al nuevo usuario \{#transfer-access-to-new-user\}

Esta opción es la más adecuada para aplicaciones que permiten compras **con o sin autenticación**, o que quieran aplicar una política de **un dispositivo por usuario**.

Adapty limita el acceso a la compra a 1 ID de cliente a la vez. El propietario del dispositivo puede reinstalar la app, iniciar y cerrar sesión, pero no puede acceder al mismo producto desde más de un ID de cliente simultáneamente.

Con esta opción activada, los perfiles anónimos (por ejemplo, un perfil que se activa cuando el usuario cierra sesión) siempre heredan el nivel de acceso del último ID de cliente activo. Esto es necesario para evitar que se pierda el acceso más adelante.

:::warning
Cuando desactivas la opción predeterminada y activas **Transfer access to new user**, Adapty no actualiza inmediatamente los niveles de acceso de los perfiles de cliente existentes.

El cambio se produce cuando un usuario genera un nuevo evento en el store: por ejemplo, renueva la suscripción o restaura sus compras.
:::

## Desactivar la compartición de acceso de pago \{#disable-paid-access-sharing\}

Esta opción **solo es adecuada** para aplicaciones con **autenticación obligatoria** o una implementación propia de gestión de acceso. En otros casos, los usuarios podrían no poder acceder a sus compras y la aplicación corre el riesgo de **no superar la revisión obligatoria del store**.

Si desactivas la compartición de acceso de pago, Adapty vincula el producto al [ID de cliente](identifying-users#set-customer-user-id-on-configuration) activo en el momento de la compra y no comparte el nivel de acceso con ningún otro perfil de cliente. Esta política permite una distribución estricta de 1 producto por usuario.

:::warning
Cuando desactivas la compartición de acceso de pago, evitas que los IDs de cliente hereden el acceso de pago. Si un ID de cliente heredó acceso de pago en el pasado, ese acceso no se puede revocar automáticamente.
:::

:::important
En situaciones de emergencia, puede que necesites [eliminar un perfil de usuario](api-adapty/operations/deleteProfile) para que el siguiente perfil disponible (identificado o anónimo) pueda reclamar su nivel de acceso.
:::

## Compartición de acceso de pago en sandbox \{#sharing-paid-access-on-sandbox\}

Puedes configurar una política de compartición de acceso de pago específica para el entorno sandbox. Cuando pruebes compras en el entorno sandbox, ten en cuenta el siguiente comportamiento:

* Apple almacena información sobre tus compras anteriores en el historial de compras de la cuenta. El SDK de Adapty también puede acceder a él.
* Si reinstala la aplicación y Adapty detecta que el producto ya ha sido comprado, el perfil activo heredará el nivel de acceso.
* Si Apple detecta una compra existente para el producto, no permitirá que realices la misma compra dos veces, aunque el perfil activo no tenga el nivel de acceso necesario.

Este comportamiento ocurre **con independencia de tu configuración de compartición de acceso de pago**. Tu app no muestra el paywall y no puedes comprar el producto. La única solución es **borrar el historial de compras de tu cuenta**. Consulta la [guía de pruebas en sandbox](test-purchases-in-sandbox) para obtener instrucciones detalladas.

## Compartición de acceso de pago en los análisis \{#paid-access-sharing-in-analytics\}

* Adapty registra las transacciones a medida que se producen. Una misma transacción puede estar asociada a más de un perfil, pero no se contabiliza más de una vez.
* Si dos o más perfiles comparten el mismo nivel de acceso, la compra se atribuye al [perfil principal](how-profiles-work#parent-and-inheritor-profiles).
* La herencia del nivel de acceso no afecta a las estadísticas de instalación. Para saber cómo Adapty contabiliza las instalaciones, puedes seleccionar una de las dos [definiciones de instalación](installs#calculation) disponibles en la página de configuración.
