Si tienes un paywall activo generando ingresos, puede que quieras probar un nuevo flow con una pequeña parte de tu audiencia antes de hacer el cambio completo.

Los [flows](adapty-flow-builder) y los paywalls son tipos de contenido distintos, por lo que cada uno vive en su propio tipo de placement — un placement de flow y un placement de paywall nunca comparten tráfico.

Esto significa que las pruebas A/B nativas de Adapty no pueden conectar ambos: una prueba A/B Regular solo compara variantes en un único placement, y una prueba A/B Crossplacement funciona únicamente entre placements de paywall. Comparar un nuevo flow con tu paywall existente es una comparación de cohortes, no una prueba nativa única.

Esta guía muestra cómo ejecutar esa comparación como un rollout gradual: divide el tráfico entre el flow y el paywall tú mismo, y aumenta la proporción del flow con el tiempo.

## Configura el rollout \{#set-up-the-rollout\}

1. **Crea y previsualiza el flow**: Crea un nuevo [flow](adapty-flow-builder) y pruébalo en un dispositivo antes de integrarlo en tu app. Para conocer los pasos completos de migración, consulta [Migrar a flows](migrate-to-flows).
2. **Crea un placement de flow independiente**: Un placement sirve un único tipo de contenido, así que crea un nuevo placement para el flow — no puedes reutilizar el ID de tu placement de paywall existente. Mantén el placement de paywall activo.
3. **Divide el tráfico en tu lado**: Usa una herramienta de remote-config o feature-flag para decidir, por usuario, qué ID de placement pasa tu app a `getFlow` — el nuevo placement de flow o el placement de paywall existente.

:::note
[Firebase Remote Config](https://firebase.google.com/docs/remote-config) es una forma habitual de hacerlo — añade un parámetro (por ejemplo, `flow_rollout_percentage`) y léelo en tu app para elegir el placement. Esto no es una función de Adapty; es una herramienta que configuras por tu cuenta, independiente del [Remote Config de Adapty](customize-paywall-with-remote-config), que configura el contenido de flows y paywalls en sí.
:::

4. **Asigna usuarios de forma persistente**: Agrupa a cada usuario una sola vez, por ejemplo hasheando su ID de usuario, para que sigan viendo la misma opción — flow o paywall — mientras cambias el porcentaje del rollout. De lo contrario, un usuario podría alternar entre ambos en cada sesión.
5. **Aumenta el porcentaje gradualmente**: Empieza con poco, por ejemplo un 10%, y auméntalo con el tiempo — 25%, 50%, 100% — a medida que confirmes que el flow funciona como se espera.

## Compara los resultados \{#compare-results\}

Como asignas cada usuario a una cohorte — flow o paywall — en lugar de que Adapty los aleatorice y rebalancee como haría una prueba A/B nativa, trata los dos placements como cohortes separadas en vez de variantes de prueba. Cada placement registra sus propias métricas, así que filtra tus [analíticas](placement-metrics) por placement para ver los números de cada cohorte.

Compara las tasas de conversión entre las dos cohortes, no los totales absolutos: a medida que aumentas el porcentaje del rollout, la cohorte del flow crece y la del paywall se reduce, por lo que sus números absolutos dejan de ser comparables aunque sus tasas sigan siéndolo.

## Una vez que hayas migrado completamente \{#once-youve-fully-switched\}

Cuando todo tu tráfico vaya al flow, retira el antiguo placement de paywall. A partir de ahí, usa una [prueba A/B Regular](ab-tests) nativa entre variantes de flow en el placement de flow para seguir optimizando — consulta [Tipos de prueba A/B](ab-test-types) para ver en qué se diferencian las pruebas Regular y Crossplacement.
