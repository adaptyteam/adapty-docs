Adapty поддерживает сценарии web2app, отслеживая веб-платежи и подписки, оформленные через [Stripe](https://stripe.com/).

Интеграция охватывает покупки, инициированные в вебе (Stripe Checkout, hosted-страницы оплаты или кастомные веб-флоу), и синхронизирует их с доступом в мобильном приложении и аналитикой.

Это полезно в следующих сценариях:

- Автоматическое предоставление доступа к платным функциям пользователям, которые совершили покупку в вебе, а затем установили приложение и вошли в свой аккаунт
- Хранение всей аналитики по подпискам в одном дашборде Adapty (включая когорты, прогнозы и остальной аналитический инструментарий)

Хотя веб-покупки становятся всё популярнее для приложений, Apple App Store разрешает альтернативные системы встроенных покупок для цифровых товаров только в США. Убедитесь, что вы не продвигаете веб-подписки внутри приложения для других стран, иначе приложение может быть отклонено или заблокировано.

Ниже описаны шаги по настройке интеграции со Stripe.

:::important
Эта интеграция посвящена отслеживанию и синхронизации веб-покупок через Stripe. Если вам нужно направлять пользователей из приложения на веб-чекаут, см. [Веб-пейволы](web-paywall).
:::

## 1\. Подключите Stripe к Adapty \{#1-connect-stripe-to-adapty\}

Интеграция в основном работает за счёт того, что Adapty получает данные о подписках из Stripe через вебхук. Поэтому нужно связать аккаунт Adapty с аккаунтом Stripe, предоставив API-ключи и указав URL вебхука Adapty в Stripe. Чтобы автоматизировать настройку вебхука, установите приложение Adapty в Stripe:

:::note
Шаги ниже одинаковы для Production и Test режимов Stripe, но для каждого из них потребуются разные API-ключи.
:::

## 4\. Обогатите веб-покупки идентификатором пользователя \{#4-enrich-purchases-made-on-the-web-with-your-user-id\}

Adapty использует вебхуки от Stripe как единственный источник информации для предоставления и обновления уровней доступа для пользователей. Но чтобы интеграция работала корректно, вы должны передавать дополнительные данные со своей стороны при работе со Stripe.

Чтобы уровни доступа были согласованы на всех платформах (веб и мобильное приложение), необходимо использовать единый идентификатор пользователя, который Adapty сможет распознать из вебхуков. Это может быть email пользователя, номер телефона или любой другой ID из используемой вами системы авторизации.

Определите, какой ID вы хотите использовать для идентификации пользователей. Затем перейдите в ту часть кода, которая инициирует платёж через Stripe, и добавьте этот ID в объект `metadata` объекта [Stripe Subscription](https://docs.stripe.com/api/subscriptions/object#subscription_object-metadata) (`sub_...`) или [Checkout Session](https://docs.stripe.com/api/checkout/sessions/create#create_checkout_session-metadata) (`ses_...`) под ключом `customer_user_id`:

```json showLineNumbers title="Stripe Metadata contents"
{'customer_user_id': "YOUR_USER_ID"}
```

Это единственное изменение в коде, которое от вас требуется. После этого Adapty будет разбирать все вебхуки от Stripe, извлекать этот `metadata` и корректно связывать подписки с вашими пользователями.

:::warning
User ID обязателен

Без него у нас нет возможности идентифицировать пользователя и предоставить ему уровень доступа в мобильном приложении.

Если вы не передаёте `customer_user_id` в `metadata`, можно настроить Adapty так, чтобы он искал `customer_user_id` в других местах: либо в поле `email` объекта Customer в Stripe, либо в поле `client_reference_id` объекта Session в Stripe.

Подробнее о настройке поведения при создании профиля читайте [ниже](stripe#profile-creation-behavior).
:::

:::note
Customer в Stripe тоже обязателен

Если вы используете Checkout Sessions, [убедитесь, что создаёте Stripe Customer](https://docs.stripe.com/api/checkout/sessions/create#create_checkout_session-customer_creation), установив `customer_creation` в значение `always`.
:::

## 5\. Предоставьте доступ пользователям в мобильном приложении \{#5-provide-access-to-users-on-the-mobile\}

Чтобы пользователи мобильного приложения, пришедшие с веба, могли получить доступ к платным функциям, просто вызовите `Adapty.activate()` или `Adapty.identify()` с тем же `customer_user_id`, который вы передали на предыдущем шаге (подробнее см. <InlineTooltip tooltip="Идентификация пользователей">[iOS](identifying-users), [Android](android-identifying-users), [React Native](react-native-identifying-users), [Flutter](flutter-identifying-users) и [Unity](unity-identifying-users)</InlineTooltip>).

## Поведение при создании профиля \{#profile-creation-behavior\}

Adapty должен привязать покупку к [профилю пользователя](profiles-crm), чтобы она была доступна в мобильном приложении — поэтому по умолчанию профили создаются при получении вебхуков от Stripe. Вы можете выбрать, что использовать в качестве идентификатора пользователя в Adapty:

1. **По умолчанию и рекомендуется:** `customer_user_id`, переданный в metadata на [шаге 4 выше](stripe#4-enrich-purchases-made-on-the-web-with-your-user-id)
2. `email` в объекте Customer в Stripe (см. [документацию Stripe](https://docs.stripe.com/api/customers/object#customer_object-email))
3. `client_reference_id` в объекте Session в Stripe (см. [документацию Stripe](https://docs.stripe.com/api/checkout/sessions/create#create_checkout_session-client_reference_id))

Настроить, какой ID использовать, можно в [App Settings → Stripe](https://app.adapty.io/settings/stripe).

:::warning
**Важно:** если конкретная транзакция из Stripe не содержит указанного ID, профиль создан не будет. Такая транзакция останется анонимной, пока не будет привязана к какому-либо профилю (например, если вы впоследствии воспользуетесь [S2S validate](api-adapty/operations/validateStripePurchase) и сообщите нам об этой транзакции вручную).

Она появится в аналитике, но не в разделах, которые считают профили (LTV, когорты, конверсии и т.д.), и вы не сможете увидеть её в ленте событий.
:::

Также есть четвёртый вариант — вообще не создавать профили, но это не рекомендуется из-за указанных выше ограничений в аналитике.

## Текущие ограничения \{#current-limitations\}

### Апгрейд, даунгрейд и пропорциональное распределение \{#upgrading-downgrading-and-proration\}

Изменения подписки — апгрейд или даунгрейд — могут приводить к пропорциональным списаниям. Adapty не учитывает эти списания в расчёте выручки. Лучше отключить эти опции вручную через дашборд Stripe. Также это можно сделать, установив атрибут `proration_behaviour` в значение `none` через Stripe API.

### Отмены \{#cancellations\}

В Stripe есть два варианта отмены подписки:

1. Немедленная отмена: подписка отменяется сразу с опцией пропорционального распределения или без неё
2. Отмена в конце периода: подписка отменяется по истечении текущего расчётного периода (аналогично встроенным подпискам в магазинах приложений)

Adapty поддерживает оба варианта, однако при расчёте выручки для немедленной отмены опция пропорционального распределения игнорируется.

### Проблемы с оплатой и льготный период \{#billing-issues-and-grace-period\}

Когда у пользователя возникают проблемы с оплатой, Adapty генерирует событие billing issue и отзывает доступ. Льготный период Stripe пока не поддерживается — это будет реализовано в будущих релизах.

### Возвраты \{#refunds\}

Adapty отслеживает только полные возвраты. Частичные возвраты и возвраты с пропорциональным распределением в настоящее время не поддерживаются.

### Уникальность идентификаторов транзакций \{#transaction-id-uniqueness\}

Adapty сопоставляет профили и транзакции, используя `store_transaction_id` и `store_original_transaction_id`. Они **должны быть уникальны** в рамках Test и Production окружений.

#### Почему это важно \{#why-this-matters\}

Если один и тот же ID транзакции существует в обоих окружениях, Adapty считает их одной транзакцией, что приводит к следующему:
- Production-покупки наследуют уровни доступа и ID продуктов из Test-окружения
- Неверные ID продуктов и окружений в ответах API
- Нарушение привязки профилей и событий подписок

#### Как обеспечить уникальность \{#how-to-ensure-uniqueness\}

ID инвойсов в Stripe могут совпадать между Test и Live окружениями. Чтобы избежать коллизий между окружениями, выберите один из подходов.

#### Вариант 1: Нумерация на уровне аккаунта с префиксами для окружений \{#option-1-account-level-numbering-with-environment-prefixes\}

Настройте префиксы отдельно для каждого окружения:

1. В Stripe Dashboard переключитесь в Test mode.
2. Перейдите в [Settings → Billing → Invoices](https://dashboard.stripe.com/settings/account/?support_details=true).
3. Установите **Invoice numbering** в **Sequentially across your account**.
4. Установите **Invoice prefix** в TEST- (или любой другой префикс, специфичный для тестового окружения).
5. Переключитесь в Live mode и повторите шаги 2–4, используя LIVE- (или любой другой префикс, специфичный для рабочего окружения) в качестве префикса.

#### Вариант 2: Нумерация на уровне пользователя \{#option-2-customer-level-numbering\}

Установите **Invoice numbering** в [**Stripe settings** -> **Billing** -> **Invoices** tab](https://dashboard.stripe.com/settings/account/?support_details=true) в значение **Sequentially for each customer (customer-level)**.

Даже при такой конфигурации, если вы удалите инвойс, Stripe может повторно использовать этот ID для новых инвойсов того же пользователя. По возможности избегайте удаления инвойсов.

## Максимум от данных Stripe \{#get-more-from-your-stripe-data\}
После интеграции со Stripe Adapty сразу готов предоставлять аналитику. Чтобы получить от данных Stripe максимум, можно настроить дополнительные интеграции Adapty для пересылки событий Stripe — так вся аналитика по подпискам окажется в одном дашборде Adapty.

:::tip
Для расширенной аналитики можно добавить `variation_id` в metadata Stripe, чтобы привязать покупки к конкретным экземплярам пейвола. Это особенно полезно при использовании собственных веб-пейволов, когда нужно отслеживать, какой именно показанный пейвол привёл к конверсии.

Обратите внимание, что `variation_id` читается только из metadata объектов Stripe Subscription (`sub_...`) и Checkout Session (`ses_...`):

```json showLineNumbers title="Stripe Metadata with variation_id"
{
  'customer_user_id': "YOUR_USER_ID",
  'variation_id': "YOUR_VARIATION_ID"
}
```
:::

Интеграции для пересылки и анализа событий Stripe:
