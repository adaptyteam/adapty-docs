Атрибуция Apple Ads (AA) поступает асинхронно после вызова `adapty.activate()`. При первом запуске она обычно ещё не получена, поэтому `getPaywall` разрешается по аудитории по умолчанию, и пользователи Apple Ads не попадают на пейвол, настроенный под AA-сегмент. Вместо того чтобы откладывать показ пейвола до получения атрибуции, покажите его сразу, а затем обновите, как только атрибуция AA будет применена — тогда пользователи Apple Ads увидят целевой вариант, а остальные не будут ждать. `AdaptyProfile.appliedAttributionSources` сообщает, когда атрибуция AA применена.
## Перед началом работы \{#before-you-start\}

Вам понадобится:
- Adapty Capacitor SDK версии **3.17.1** или выше.
- Настроенная интеграция Apple Ads для приложения в Adapty. См. [Apple Ads](apple-search-ads).
## Как это работает \{#how-it-works\}

После `adapty.activate()` SDK в фоновом режиме запрашивает данные атрибуции Apple Ads у Apple и передаёт результат в бэкенд Adapty. Когда AA становится активным источником атрибуции для профиля, SDK доставляет обновлённый `AdaptyProfile` в ваш слушатель `onLatestProfileLoad`, в массиве `appliedAttributionSources` которого появляется `'apple_search_ads'`.

Это позволяет загружать пейвол в два шага:
1. Вызовите `getPaywall` сразу. Поскольку атрибуция ещё не применена, Adapty обрабатывает запрос относительно аудитории по умолчанию, и пользователь видит пейвол немедленно.
2. Когда появится `'apple_search_ads'`, вызовите `getPaywall` снова. Adapty теперь обрабатывает запрос относительно аудитории Apple Ads и возвращает целевой пейвол, который заменяет первый.

`appliedAttributionSources` может быть пустым или отсутствовать. Это означает одно из двух:

- атрибуция Apple Ads ещё не обработана для этого профиля, или
- атрибуция не поступила вовсе.
При любом раскладе шаг 1 безопасен — Adapty обрабатывает запрос с учётом аудитории, соответствующей текущему состоянию профиля, как правило это аудитория по умолчанию. Шаг 2 выполняется только после того, как `'apple_search_ads'` появится в атрибуции.

:::important
При каждом последующем запуске в кешированном профиле уже содержится `'apple_search_ads'` в `appliedAttributionSources`, поэтому первый же `getPaywall` сразу возвращает пейвол, сегментированный по Apple Ads, — никакого повторного запроса или видимого изменения не происходит. Двухшаговое флоу актуально только при первом запуске, пока атрибуция ещё не получена.
:::
## Реализация \{#implementation\}

Покажите пейвол сразу, а затем слушайте событие `'apple_search_ads'` и обновляйте пейвол при его получении.
1. **Активируйте SDK.** См. [Установка и настройка Capacitor SDK](sdk-installation-capacitor).
2. **Загрузите и покажите пейвол** с помощью `getPaywall` как обычно — не блокируйте выполнение на ожидании атрибуции.
3. **Подпишитесь на обновления профиля** через `adapty.addListener('onLatestProfileLoad', …)` и отслеживайте появление `'apple_search_ads'`. Когда оно появится, снова запросите пейвол и покажите обновлённый. Если слушатель ещё не настроен, см. [Подписка на обновления подписки](capacitor-check-subscription-status#listen-to-subscription-updates):
```typescript
const handle = await adapty.addListener('onLatestProfileLoad', async ({ profile }) => {
  if (!profile.appliedAttributionSources?.includes('apple_search_ads')) return;
  const targeted = await adapty.getPaywall({ placementId });
  // present the targeted paywall in place of the first one
});

// Call handle.remove() after the upgrade, or after a timeout (see below).
```

4. **Остановите прослушивание после таймаута.** Большинство пользователей никогда не получают атрибуцию Apple Ads, поэтому удаляйте слушателя через несколько секунд вместо того, чтобы держать его активным на протяжении всей сессии. Настройте [резервный пейвол](capacitor-use-fallback-paywalls) для плейсмента, чтобы пользователь всегда что-то видел при неудачном запросе.
## Полный пример \{#complete-example\}

Функция ниже немедленно показывает пейвол, а затем заменяет его на вариант, сегментированный по Apple Ads, если атрибуция применяется в течение `timeoutMs`. Каждый пейвол передаётся через колбэк `onPaywall`, который можно подключить к UI — никто не ждёт, а пользователи Apple Ads получают целевой пейвол сразу после получения атрибуции.

`waitForAppleAdsAttribution` завершается успешно, как только атрибуция применена, или выбрасывает ошибку по истечении таймаута. `loadPaywallWithAppleAdsUpgrade` запускает это ожидание до первого запроса, показывает пейвол для аудитории по умолчанию, а затем повторно запрашивает пейвол после получения атрибуции:
```typescript title="loadPaywallWithAppleAdsUpgrade.ts"

const APPLE_ADS_SOURCE = 'apple_search_ads';

function hasAppleAdsAttribution(profile: AdaptyProfile): boolean {
  return profile.appliedAttributionSources?.includes(APPLE_ADS_SOURCE) ?? false;
}

/**
 * Resolves once Apple Ads attribution is applied to the profile, or rejects
 * after `timeoutMs`. Removes the listener and clears the timer on every path.
 */
async function waitForAppleAdsAttribution(timeoutMs: number): Promise<void> {
  let onAttributionApplied!: () => void;
  const attributionApplied = new Promise<void>(resolve => {
    onAttributionApplied = resolve;
  });

  const handle = await adapty.addListener('onLatestProfileLoad', ({ profile }) => {
    if (hasAppleAdsAttribution(profile)) onAttributionApplied();
  });

  // Events aren't replayed, so read the current profile once too. The listener
  // is already active, so an update firing in between can't be missed.
  adapty
    .getProfile()
    .then(profile => {
      if (hasAppleAdsAttribution(profile)) onAttributionApplied();
    })
    .catch(() => {
      // Offline with no cached profile — the listener or timeout settles the race.
    });

  let timer: ReturnType<typeof setTimeout> | undefined;
  const timedOut = new Promise<never>((_, reject) => {
    timer = setTimeout(() => reject(new Error('Apple Ads attribution timed out')), timeoutMs);
  });

  try {
    await Promise.race([attributionApplied, timedOut]);
  } finally {
    clearTimeout(timer);
    await handle.remove();
  }
}

/**
 * Shows a paywall immediately, then upgrades it to the Apple-Ads-segmented
 * variant if attribution is applied within `timeoutMs`. Each paywall is reported
 * through `onPaywall`: the first one renders right away, and Apple Ads users get
 * the targeted one as soon as attribution lands. Rejects only if the first fetch
 * fails — an attribution timeout is the normal non-Apple-Ads outcome, not an error.
 */
export async function loadPaywallWithAppleAdsUpgrade(
  placementId: string,
  timeoutMs: number,
  onPaywall: (paywall: AdaptyPaywall) => void,
): Promise<void> {
  // Start watching for attribution before the first fetch, so an update that
  // lands while it's loading isn't missed. Don't await it yet — the first
  // paywall must not wait.
  const attributionApplied = waitForAppleAdsAttribution(timeoutMs);
  attributionApplied.catch(() => {});

  // Show a paywall now. With no attribution yet, this is the default audience.
  onPaywall(await adapty.getPaywall({ placementId }));

  try {
    await attributionApplied;
    // Attribution applied — re-fetch so Apple Ads users get the targeted paywall.
    onPaywall(await adapty.getPaywall({ placementId }));
  } catch {
    // Attribution didn't arrive in time — keep the paywall already shown.
  }
}
```

:::note
События плагина Capacitor не воспроизводятся повторно для слушателя, который подключается после их возникновения. Именно поэтому `waitForAppleAdsAttribution` одновременно подписывается на `onLatestProfileLoad` и однократно вызывает `getProfile()`: слушатель перехватывает атрибуцию, которая поступает в течение текущего запуска, а `getProfile()` — атрибуцию, уже применённую в предыдущем запуске.
:::

Подключите это к месту, где вы показываете пейвол:
```typescript
loadPaywallWithAppleAdsUpgrade('YOUR_PLACEMENT_ID', 5000, paywall => {
  // Present the paywall. If it's already on screen, replace it with this one.
}).catch(() => {
  // The first load failed (for example, no network) — show a fallback paywall.
});
```

При первом запуске пользователь Apple Ads на долю секунды видит пейвол по умолчанию, прежде чем он заменяется. Если вы показываете пейволы с помощью Paywall Builder, решите, приемлемо ли повторное отображение, или применяйте замену до показа пейвола. Для `timeoutMs` достаточно нескольких секунд — атрибуция, если она ожидается, приходит вскоре после запуска.
Если ваше приложение уже слушает `onLatestProfileLoad` для других целей (например, [проверки статуса подписки](capacitor-check-subscription-status#listen-to-subscription-updates)), менять ничего не нужно. `adapty.addListener` поддерживает несколько независимых слушателей, поэтому этот добавляется самостоятельно, не затрагивая остальные.
