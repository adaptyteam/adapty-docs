Серверный API v2 Adapty предоставляет новые возможности и улучшения для более эффективного управления уровнями доступа, транзакциями и профилями пользователей.
## Зачем переходить на новую версию? \{#why-migrate\}

Вторая версия Server-Side API даёт больше гибкости и возможностей при работе с Adapty:
- **Раздельное управление уровнями доступа**: назначайте уровни доступа пользователям без привязки к деталям транзакции — удобно для компенсаций, бонусов и других нетранзакционных сценариев.
- **Запись разовых покупок**: логируйте транзакции для расходуемых покупок, передавая поля, специфичные для таких продуктов.
- **Расширенные детали транзакции**: добавляйте к транзакциям больше данных — возвраты, проблемы с оплатой, причины отмены, продления и многое другое.
- **Обновление профиля**: вместо простого добавления атрибутов можно обновлять профиль пользователя целиком. Например, добавить метаданные об установке или отключить внешнюю аналитику при необходимости.
Хотя v1 всё ещё поддерживается, мы рекомендуем перейти на v2 — там больше возможностей и актуальные обновления.
## Обзор изменений \{#changes-overview\}
| Изменение | Необходимое действие |
| ----------------------------------- | ------------------------------------------------------------ |
| **Базовый URL и эндпоинт** | Базовый URL и все эндпоинты изменились. Обновите конфигурацию, как описано в разделе о конечных точках запросов |
| **Заголовки запроса** | <ol><li> Добавьте `adapty-profile-id` или `adapty-customer-user-id` в качестве заголовка.</li><li> Добавьте новый заголовок `adapty-platform`.</li></ol> |
| **Структура запросов и ответов** | Измените параметры согласно описанию для каждого запроса и обновите интеграцию для обработки [новых форматов ответов](api-responses). |
| **Изменённое управление уровнями доступа** | Старый запрос [Prolong/grant a subscription for a user](server-side-api-specs-legacy#prolonggrant-a-subscription-for-a-user) теперь разделён на три: <ul><li> [Set Transaction](api-adapty/operations/setTransaction): добавление деталей транзакции с предоставлением доступа.</li><li> [Grant Access Level](api-adapty/operations/grantAccessLevel): добавление или продление доступа без транзакции.</li><li> [Revoke Access Level](api-adapty/operations/revokeAccessLevel): сокращение или отзыв доступа без транзакции.</li></ul> |

### Шаг 2. Изменение запросов на управление транзакциями и уровнями доступа \{#step-2-change-transaction-and-access-management-requests\}

В версии 1 вы использовали:

- Запрос [Продление/предоставление подписки пользователю](server-side-api-specs-legacy#prolonggrant-a-subscription-for-a-user): для записи транзакции и предоставления или сокращения уровня доступа.
- Запрос [Отзыв уровня доступа](server-side-api-specs-legacy#revoke-subscription-from-a-user): для немедленного отзыва доступа.

Теперь они заменены тремя отдельными запросами, чтобы разграничить добавление транзакций и управление уровнями доступа:
1. **[Предоставить уровень доступа](api-adapty/operations/grantAccessLevel):** Используйте этот запрос, чтобы продлить уровень доступа без привязки к транзакции.
2. **[Отозвать уровень доступа](api-adapty/operations/revokeAccessLevel):** Чтобы немедленно отозвать или сократить доступ.
3. **[Установить транзакцию](api-adapty/operations/setTransaction):** Используйте этот запрос, чтобы добавить детали транзакции в Adapty с уровнями доступа.

#### Шаг 2.1. Как предоставить уровень доступа \{#step-21-how-to-grant-access-level\}

:::info

Подробное описание см. в запросе [Предоставить уровень доступа](api-adapty/operations/grantAccessLevel).

:::
В версии 1 для предоставления доступа использовался запрос [Prolong/grant a subscription for a user](server-side-api-specs-legacy#prolonggrant-a-subscription-for-a-user). Теперь вы можете предоставить доступ с помощью запроса [Grant access level](api-adapty/operations/grantAccessLevel), не указывая детали транзакции.

- **Endpoint:** `https://api.adapty.io/api/v2/server-side-api/grant/access-level/`
- Параметры, которые сохранились:
  - **access_level_id**: Ранее передавался в endpoint. Обязательный.
  - **starts_at**: Теперь может быть null.
  - **expires_at**: Необязательный для пожизненного доступа и может быть null.

#### Шаг 2.2. Как отозвать или сократить уровень доступа \{#step-22-how-to-revoke-or-shorten-access-level\}

:::info

Подробное описание см. в запросе [Revoke access level](api-adapty/operations/revokeAccessLevel).

:::

В версии 1 для немедленного отзыва доступа использовался запрос [Revoke access level](server-side-api-specs-legacy#revoke-subscription-from-a-user), а для сокращения — запрос [Prolong/Grant a Subscription for a User](server-side-api-specs-legacy#prolonggrant-a-subscription-for-a-user). Теперь для обоих действий можно использовать запрос [Revoke access level](api-adapty/operations/revokeAccessLevel).
- **Endpoint:** `https://api.adapty.io/api/v2/server-side-api/purchase/profile/revoke/access-level/`
- Параметры:
  - **access_level_id**: Обязательный.
  - **expires_at**: Nullable, если только доступ не отзывается немедленно.
